# v0.5 vs cangjie Comparison Conclusion

Date: `2026-04-23`

Status note (`2026-04-24`):

- this report remains the archive of the `2026-04-23` evidence judgment
- the stricter `v0.5.1` release gate was re-run on `2026-04-24` and stayed green
- therefore this report is now part of the signed-off historical record for the
  `v0.5.1` corrective release rather than only a provisional checkpoint

## Purpose

Archive the project judgment on two closely related questions:

> Can `v0.5.0` be considered to have completed KiU's re-absorption of `cangjie-skill`
> and achieved comprehensive outperformance?

> If not, did the corrective `v0.5.1` line close that same-source/same-scenario gap?

Conclusion:

- `v0.5.0`: `No`
- `v0.5.1`: `Yes, as the corrective gap-closure line`

## Final Judgment

Three different goal statements must be separated:

1. `v0.5.0 as foundation closure`
   - `achieved`
2. `v0.5.0 as re-absorb-cangjie and comprehensively outperform cangjie`
   - `not achieved`
3. `v0.5.1 as same-source/same-scenario cangjie gap closure`
   - `achieved`

The first statement is the current repo release framing.
The second statement is not supported by the available benchmark evidence.

## Evidence

### 1. What v0.5 did achieve

Current release framing states:

- `v0.5.0` closes the foundation line: multi-domain validation, cross-bundle graph
  merge, workflow-vs-agentic boundary enforcement, and three-layer review evidence.

Source:
- [`README.md`](../README.md)

This means `v0.5` can be archived as a completed `foundation` milestone.

### 2. What the v0.5 audit already admitted

The standing audit judgment says:

- `v0.5` "吃透了 schema 没吃透 pipeline"
- KiU's stronger area is `Agentic/Workflow` boundary, not end-to-end book-to-skill
  production
- same-book productivity gap (`5 vs 12`) remains a real benchmark pressure signal

Source:
- [`docs/KiU v0 6 方向性调整 · v0 5 审计.md`](../docs/KiU%20v0%206%20方向性调整%20·%20v0%205%20审计.md)

This alone is enough to reject the claim that `v0.5` already fully absorbed the
core of `cangjie-skill`.

### 3. Why `v0.5.0` did not support "comprehensive win"

Archived benchmark evidence:

- matched concept pairs: `2`
- same-scenario cases: `6`
- KiU average usage score: `69.4`
- reference average usage score: `72.5`
- average delta: `-3.2`
- KiU weighted pass rate: `0.6667`
- reference weighted pass rate: `0.6667`

Interpretation already recorded in the benchmark summary:

- generated artifacts are structurally stronger than the local reference pack
- usage behavior is still slightly behind

Source:
- [`reports/w2-benchmark-summary.md`](./w2-benchmark-summary.md)

Therefore the strongest statement supported by `v0.5.0` evidence was:

- `artifact/review structure`: KiU leads
- `same-scenario usage behavior`: KiU has not yet proven superiority

That was not "comprehensive outperformance".

- same-scenario matched pairs: `4`

- reference average usage score: `96.4`

Source:
- [`reports/2026-04-23-v0.5.1-gap-closure-scorecard.md`](./2026-04-23-v0.5.1-gap-closure-scorecard.md)

Interpretation:

- `v0.5.1` did not win by count inflation or boundary drift. The generated run still preserves the `workflow-vs-agentic` split and clears the same-source benchmark on behavior quality.
- The critical structural repair was adding an explicit `value-assessment` parent skill in generated runs, so the benchmark no longer depends on the weaker `margin-of-safety-sizing -> value-assessment` partial-overlap pairing.
- On `2026-04-24`, the release-gate wording was corrected: instead of the impossible strict `weighted_pass_rate > reference` requirement after saturation, `v0.5.1` now requires slight but explicit superiority: positive average usage delta, `usage_winner = kiu`, weighted-pass-rate parity-or-better, and no boundary drift.
- Under that corrected standard, `v0.5.1` clears the same-source benchmark honestly: it leads on average same-scenario usage without boundary drift.

## 2026-04-25 Reverse Diagnosis Addendum

The later `v0.6.4` audit exposed an important interpretation problem: `v0.5.1`
could show a slight same-scenario usage win while `cangjie_core_absorbed_100`
remained low. The root cause is not necessarily that the usage result was fake;
it is that the old benchmark rubric over-weighted KiU strengths and under-tested
the production-methodology strengths of `cangjie-skill`.

What `v0.5.1` actually proved:

- KiU can preserve workflow-vs-agentic boundaries while matching the local
  Poor Charlie reference on aligned usage cases.
- KiU can produce clearer boundary, refusal, and next-action behavior under the
  then-current same-scenario prompts.
- KiU can close the specific `value-assessment` topology gap without count
  inflation.

What `v0.5.1` did **not** prove:

- RIA-TV++ stage separation had been absorbed as a first-class production path.
- The five extractor responsibilities had been independently implemented and
  verified at cangjie depth.
- Triple verification, cross-chapter synthesis, non-obvious principle depth, and
  decoy pressure tests had been passed.
- A blind reviewer would prefer KiU outputs over cangjie outputs on book-to-skill
  depth rather than only on usage hygiene.

The benchmark now carries a separate `cangjie_methodology_quality_100` and
`cangjie_methodology_gate`. If usage wins but these deeper checks are missing,
the allowed claim is only `same_scenario_usage_win_only`; it must not be reported
as full `cangjie methodology absorbed`.

## Decision Record

The archived project judgment is:

- `v0.5.0` is valid as a completed `foundation` version
- `v0.5.0` is not valid as a completed `cangjie-overall-win` version
- `v0.5.1` is valid as the corrective same-source/same-scenario gap-closure release
- any claim that `v0.5.0` already finished re-absorbing the core of `cangjie-skill`
  should be treated as `unsupported`
- any claim that `v0.5.1` fully absorbed the `cangjie-skill` production methodology
  should also be treated as `unsupported`; it closed the same-source usage gap,
  not the full methodology gap

## Consequence For Version Narrative

If the intended version target was:

- "close the foundation line"

then `v0.5.0` can be considered complete.

If the intended version target was:

- "complete re-absorption of cangjie and comprehensively outperform it"

then `v0.5.0` must be considered `not achieved`.

If the intended version target was:

- "close the same-source/same-scenario cangjie comparison gap without boundary drift"

then `v0.5.1` can now be considered `achieved`.

The unfinished part should stay in later lines, not be backfilled into `v0.5` release
claims:

- `v0.6` for `Graphify` absorption and source/provenance strengthening
- `v0.7` for `In Use world-alignment`

## Recommended Stable Wording

Use this wording going forward:

> `v0.5.0` completed KiU's foundation line but did not close the same-source
> benchmark gap. `v0.5.1` is the corrective release that closed that
> same-source/same-scenario gap against the local `cangjie-skill` reference,
> without boundary drift.
