# v0.7.1 Audit Response Addendum

This addendum responds to the v0.7.1 review findings using the released `v0.7.1` tag evidence. It does not change the v0.7.1 release claim and does not reopen the release tag.

## Scope

The review used the L1/L2/L3 frame:

- L1 mechanism: whether the world-alignment mechanisms exist and preserve the source/world boundary.
- L2 runtime: whether those mechanisms are exercised by release evidence.
- L3 value: whether value metrics and comparison evidence are visible enough to audit.

The addendum only clarifies evidence visibility, metric interpretation, and follow-up constraints for v0.7.2.

## Finding E: Same-Book Cangjie Delta Narrowing

Review finding E noted that the v0.7.1 same-book cangjie reference delta narrowed to `+0.5`, compared with stronger v0.6.x reference deltas.

Project response: the narrowing is real as an observed result, but it cannot be directly attributed to world alignment from the available evidence alone.

The v0.7.1 synthesis reports:

| Metric | Result |
| --- | ---: |
| Matched pairs | `4` |
| Scenario count | `37` |
| KiU average usage | `97.0` |
| Reference average usage | `96.4` |
| Average delta | `+0.5` |
| KiU weighted pass rate | `1.0` |
| Reference weighted pass rate | `1.0` |
| Weighted pass-rate delta | `0.0` |

Important boundaries:

- This is a `same-book-reference` comparison, not strict same-source-data generation.
- The v0.7.1 scenario count is `37`; earlier comparisons used different scenario sets and sometimes different benchmark scope.
- The cangjie methodology gate remains false for this benchmark path because the path is `source_bundle_regeneration`, not a raw-book cold-start methodology closure run.
- A cross-version delta cannot by itself prove that world alignment helped or hurt the benchmark.

Working hypotheses to test later:

- Scenario-set shift: the added cases may cover harder principle-heavy or edge prompts.
- Runtime-gate scoring: ask/refuse/caveat behavior may reduce direct-answer proxy usage while improving misuse prevention.
- Proxy brittleness: Poor Charlie remains weak for some principle-heavy high-risk and edge prompts.

Follow-up rule for v0.7.2: release evidence must include `value_delta_and_regression_attribution`, separating same-run on/off ablation from cross-version or same-book reference comparisons.

## Finding A: Mechanism Count Alignment

Review finding A noted that `candidate_arbitration_count=31`, while `accepted_pressure_count=20` and `rejected_pressure_count=5`, leaving six candidates unexplained.

Project response: the concern is valid. The v0.7.1 gate evidence reports mechanism counters but does not state whether each counter is per-skill, per-candidate, or per-gate. The likely issue is mixed counter grain rather than a runtime failure, but the evidence should not require readers to infer this.

Follow-up rule: future mechanism-count evidence must include counter grain and a complete residual classification. If `arbitrated = accepted + rejected + other`, the `other` bucket must be named.

## Finding B: Missing `light` Intervention Level

Review finding B noted that the intervention distribution contains `minimal`, `moderate`, and `strong_gate`, but no `light` examples.

Project response: the observation is valid, but it is not release-blocking for v0.7.1. The sample set is small. It proves the three observed levels ran; it does not prove the `light` threshold is dead.

Follow-up rule: v0.7.2+ should track intervention-level coverage. If broader samples still produce no `light` cases, the threshold or the level itself should be reviewed.

## Finding F: Real Samples Versus Fixture Samples

Review finding F noted that `samples_passed=4` includes one synthetic fixture.

Project response: the concern is valid. The v0.7.1 release gate still covers the required three real samples, but the count should be clearer.

Clarified count:

| Sample Type | Count | Items |

| Real bundle | `3` | Poor Charlie, Effective Requirements, Financial Statement |
| Fixture | `1` | `no_web_refuse_fixture` |

Follow-up rule: future release evidence should report real samples and fixtures separately.

## Preserved Evidence-Honesty Practice

v0.7.1 deliberately did not force a fake `partial_apply` verdict only to satisfy the stretch target. This practice should be retained:

```text
Do not manufacture verdict diversity or metric coverage just to pass a stretch target.
```

This rule is a project evidence-honesty norm, not a one-off release note.

## v0.7.2 Carry-Forward Constraints

The v0.7.2 live fact verification line should inherit these constraints:

- Release-facing evidence must expose actuals, thresholds, statuses, and evidence paths in one place.
- Live-web value claims need same-run on/off or supported/unsupported attribution, not only cross-version comparisons.
- Same-book reference comparisons must remain labeled as same-book unless source-data parity is proven.
- Mechanism counters must state their grain and residual buckets.
- Real samples and fixtures must be counted separately.
- Evidence honesty remains stronger than metric cosmetics; do not force verdicts or cases to satisfy stretch targets.
