# KiU v0.6 规划意见与工单计划（基于 v0.5 复核）

## 1. 结论摘要

基于对已发布 `kiu-skills-v0.5.0` 的复核，`v0.6` 的方向不应该是“重新定义 foundation”，而应该是：

1. 以 `v0.5` 为稳定底座，不重写已经成立的能力。
2. 把真正缺失的能力收敛为一条主线：`raw book -> extraction -> graph -> distillation -> routing -> review`。
3. 用端到端产出质量，而不是 schema 完整度，作为 `v0.6` 的第一性目标。

## 2. 对 v0.5 审计的修正意见

### 2.1 关于 pipeline

`v0.5` 不能被描述为“pipeline = 0”。

更准确的说法是：

- `v0.5` 已经有 **seed/refinement candidate pipeline**
- `v0.5` 缺的是 **raw-book extraction pipeline**

也就是说，`v0.6` 的任务不是“从零开始发明 pipeline”，而是补齐从原书到 source bundle / graph seeds 的上游生产段。

### 2.2 关于 Agentic / Workflow

审计里对这条线的评价方向基本对，但低估了 `v0.5` 的完成度。

`v0.5` 已经不只是：

- `routing_rules`
- `workflow_candidates/`
- 评分时计入 `workflow_boundary_factor`

它还会实际生成：

- `workflow.yaml`
- `CHECKLIST.md`

所以 `v0.6` 不该重做“边界原则”，而应该做两件更具体的事：

- 让 `workflow_certainty` 的判定尽量从抽取证据中长出来，而不是长期依赖人工 override
- 增加 workflow 候选的数量与分布，证明边界机制可规模化

### 2.3 关于 Graphify / cangjie 对标

这两条参照线可以继续用，但口径要更严格：

- `Graphify 25%`、`cangjie 45%` 这类百分比只能当内部判断，不应当成“可验证事实”
- `5 vs 12` 可以作为 benchmark 目标，但 `12` 这半句如果没有在本 repo 内部复验，不应当写成既成事实

`v0.6` 计划里建议改成：

- `Graphify-style provenance graph` 是目标形态
- `cangjie-style book-to-skill throughput` 是 benchmark
- 不再写不可审计的完成度百分比

## 3. v0.6 的规划意见

## 3.1 不要重写的东西

`v0.5` 里以下能力已经成立，应视为回归基线：

- 多 domain bundle 结构
- 跨 bundle graph merge
- Agentic / Workflow 路由
- 三层审查评分
- production quality gate

`v0.6` 开发时，这些模块默认只允许“向前兼容扩展”，不做推倒重写。

## 3.2 v0.6 的真实主目标

`v0.6` 的核心目标应定义为：

> 从单本原书文本出发，稳定产出带出处信息的 graph、可筛选的 skill/workflow 候选，以及可复核的三层审查结果。

这意味着 `v0.6` 的优先级顺序应是：

1. 原书 ingestion / chunking
2. extraction outputs 的中间表示
3. provenance-rich graph schema
4. distillation / routing / review 接线
5. graph report / clustering / 辅助可视化

也就是说，Leiden 聚类和 `GRAPH_REPORT.md` 是有价值的，但不应抢在 extraction IO 合同之前成为 W1 主目标。

## 3.3 v0.6 的发布门槛建议

建议把 `v0.6` 的验收门槛改成更可审计的版本：

- 单本《穷查理宝典》端到端跑通
- 至少产出 `10-15` 个 `good` 或以上的 candidate skills
- 至少产出 `2` 个可交付 workflow candidates
- graph 中每个 candidate 都有 provenance 字段
- 端到端结果可落 `reports/three-layer-review.json`

相比“必须 15+”，这个门槛更适合第一版生产管线，因为它既保留了压力，也避免为了凑数量牺牲质量。

## 3.4 v0.6 的外部参照评分卡

后续版本评估不应只看“KiU 自己有没有进步”，还要看是否真正吸收了两个参照项目的核心。

建议从 `v0.6` 开始固定一张内部评分卡，作为阶段评审模板：

### Graphify 核心参照

只看内核，不看外围：

- `graph.json schema`
  - 是否具备 `source_file`
  - 是否具备 `source_location`
  - 是否具备边级 `confidence`
  - 是否具备边级 `extraction_kind`
- 三态边标记
  - 是否明确区分 `EXTRACTED / INFERRED / AMBIGUOUS`
- 两段式提取架构
  - 是否有 deterministic pass
  - 是否有 LLM pass
  - 是否能保留审计轨迹
- 社区检测
  - 是否从图自动生成 communities，而不是长期手工维护
- `GRAPH_REPORT.md`
  - 是否能输出 god nodes / communities / surprising connections / suggested questions

### cangjie-skill 核心参照

只看方法论与生产内核：

- `RIA-TV++` 风格的分阶段生产设计
- 五类 extractor 的明确分工
  - framework
  - principle
  - case
  - counter-example
  - term
- 三重验证规则
  - 跨域佐证
  - 预测力
  - 独特性
- 元 skill / book-to-skill 执行规范
- 模板文件体系
  - `SKILL.md`
  - `INDEX.md`
  - `BOOK_OVERVIEW.md`
  - `test-prompts.json`

### 建议评分方式

后续每个版本都按三组分开打：

1. `KiU foundation retained`
   - merge / routing / review / quality gate 是否保持
2. `Graphify core absorbed`
   - provenance graph / 三态边 / 两段式提取 / clustering / graph report
3. `cangjie core absorbed`
   - 分阶段 pipeline / extractor 设计 / 三重验证 / book-to-skill 模板

这样做的好处是：

- 不会把外围生态误算成“核心能力”
- 不会把不可验证的主观百分比当作事实
- 可以直接看到每个版本究竟是在补 graph 核心，还是在补蒸馏核心

## 4. 建议的四周拆解

### W1：基线冻结 + provenance 合同

目标：把 `v0.5` 作为不可退化底座固定下来，同时定义 `v0.6` 的 source/extraction/graph 合同。

交付：

- attribution 文件就位
- `v0.5` 回归基线命令固定
- `graph schema v0.2` 定稿
- source chunk / extraction output schema 定稿
- 当前两个 bundle 的 graph 迁移脚本可跑

### W2：提取前半段

目标：从原书文本走到“可用的 extraction outputs”。

交付：

- source ingestion
- chunking / section map
- deterministic extraction skeleton
- LLM extraction interface
- provenance-rich interim artifacts

### W3：蒸馏与分流

目标：把 extraction outputs 接到 KiU 现有 candidate / routing / review 管线上。

交付：

- extracted graph -> candidate seeds
- candidate 去重 / 过滤 / ranking
- Agentic / Workflow 自动分流
- workflow 候选数量从 `1` 提升到 `>=2`

### W4：端到端 benchmark

目标：拿《穷查理宝典》做第一轮真实生产 benchmark。

交付：

- 完整 graph
- 10-15 个 `good+` candidate skills
- 2 个以上 workflow candidates
- 三层审查报告
- 对照 benchmark 的质量总结

## 6. 对 W1 排序的建议

建议顺序：

1. `KIU-611` Attribution
2. `KIU-610` v0.5 baseline freeze
3. `KIU-612` source ingestion schema
4. `KIU-613` graph schema v0.2
5. `KIU-614` graph migration
6. `KIU-615` extraction skeleton CLI
7. `KIU-616` examples regression set

原因：

- 先把法律和基线锁住
- 再定义输入输出合同
- 再迁移现有图
- 最后才开始搭 extraction 入口

这比“先做 Leiden / GRAPH_REPORT”更贴近 `v0.6` 的真实主矛盾。

## 7. 我的总判断

`v0.6` 应该被定义为：

> **在 `v0.5` 已成立 foundation 上，自研 raw-book production pipeline 的第一版。**

如果按这个定义推进，计划会更稳，也更符合这次复核后的事实。
