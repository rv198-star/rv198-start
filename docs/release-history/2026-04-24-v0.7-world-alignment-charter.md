# KiU v0.7 World Alignment Charter

Date: `2026-04-24`

## v0.6.x Prerequisite

`v0.7` implementation is gated on the v0.6.x cold-start production boundary. Before world-alignment work starts, KiU must be able to distinguish `raw_book_no_seed_cold_start` from `source_bundle_regeneration` in generated-run provenance, review output, and reference benchmark evidence.

This prevents world-alignment scores from masking an unproven book-to-skill production path. The prerequisite is tracked in `docs/2026-04-24-v0.6.3-cold-start-production-plan.md` and backlog tickets `KIU-672` through `KIU-675`.

## Version Name

`KiU v0.7 · 学以致用 · Basic Isolated World Alignment`

中文项目名：`学以致用`。

中文定位：`隔离增强的基础真实世界对齐层`。

## Core Goal

`v0.7` adds an isolated world-alignment layer over source-faithful KiU skills.

The governing principle is `isolation enhancement`, not `fusion rewrite`: world alignment can add contextual gates, caveats, pressure tests, and decision-path explanations, but it cannot silently rewrite the source-derived skill itself.

The goal is to make generated skills more useful under realistic use pressure while preserving:

- source fidelity
- evidence honesty
- workflow-vs-agentic boundary discipline
- auditable decision paths
- scalable system-level production quality

The version does not make external world information a hidden upstream generator. It introduces a reviewer / gate / explainer layer that can pressure-test and contextualize source-derived skills without silently rewriting them.

`v0.7` must preserve an original-source-only mode for users who want to use the book/source-derived skill without world alignment. This mode is a compatibility requirement, not a niche exception.


## Goal Tier Adjustment

`v0.7.0` uses a developable-goal release model. The short-term target is to improve
KiU's internally verifiable system quality, not to prove external human preference.
External blind review, real user preference, live-world factual validation, and
domain-expert review are moved to mid/late-stage condition-dependent validation.
They remain important, but they are not blockers for the v0.7.0 foundation release.

Short-term release goals:

- preserve source fidelity while adding isolated world-alignment artifacts
- keep world alignment auditable and optional
- prevent no-web current-fact hallucination through explicit gates
- improve internal usage regression beyond shallow smoke tests
- verify cross-sample stability with repeatable local runs
- state claim boundaries without upgrading internal evidence into external validation

Condition-dependent future goals:

- human or domain-expert blind preference review
- real user task feedback
- live web / current-fact verification
- multi-model external review
- long-running production usage data

This adjustment exists because unavailable external validation should not stop
foundation development. It only limits what the release may claim.

## Non-Goal

`v0.7` is not a general runtime agent, web-search product, multi-world debate system, multi-stance adjudicator, or external fact-ingestion pipeline.

The first release explicitly defers live web search, real-time fact verification, multi-world / multi-perspective modeling, and complex validity lifecycle management. Investment and financial-planning scenarios may create a strong future need for live checks, but v0.7.0 only builds the isolated artifact structure and application gate needed to add that capability later without contaminating source-faithful skills.

## Alignment With `AGENTS.md`

### External Reference Boundary

World context is not source evidence.

External reality, current facts, user-provided situation data, benchmark references, or simulated pressure cases may be used as world-alignment context, but they must not become hidden source anchors for the default skill-generation path.

Allowed effects:

- add usage caveats
- add realistic pressure tests
- add temporal context requirements
- gate application when current context is missing
- explain decision paths

Forbidden effects:

- rewrite source-derived `SKILL.md` rationale
- replace book/source anchors
- invent authorial claims
- silently widen a skill beyond the source-supported boundary
- blend world context into source claims without a visible boundary
- make world alignment mandatory when the user requests original-source-only output
- ingest external final skill packs as default production inputs

### Workflow-vs-Agentic Boundary

World alignment must not improve apparent usefulness by collapsing deterministic workflow candidates into thick skills.

If a realistic use case only needs a stable procedure and explicit context, it remains a workflow candidate. If it requires judgment-rich interpretation, context assembly, or boundary arbitration, it may be evaluated by an agentic skill or by the world-alignment gate.

Release review must keep scoring workflow-vs-agentic boundary quality explicitly.

### System Efficiency Over Local Advantage

`v0.7` should favor repeatable, auditable, average-case improvement over handcrafted showcase examples.

World alignment should become a system layer with schemas, gates, and review scores, not a manual rewrite pass that produces a few locally excellent but non-scalable artifacts.

## Workflow-vs-Agentic Construction Rule

World context must not be produced as a static template. The construction path is split into two roles:

- `workflow`: parse skill metadata, classify temporal sensitivity, derive auditable candidate pressure dimensions, and enforce isolation/no-web fields.
- `agentic`: choose the relevant pressure dimensions, state an application hypothesis, critique genericness and hallucination risk, and revise until the context is specific enough or the loop reaches its cap.

The v0.7.0 implementation may use a deterministic `workflow_plus_agentic_proxy` loop, but the artifact must expose the loop result through `pressure_dimensions`, `world_hypothesis`, `why_this_matters`, `user_context_questions`, `deepening_rounds`, and `world_context_depth_score`.

This rule exists because a generic sentence such as "organizational practices and tooling may change application" proves only structure and isolation. It does not prove useful world modeling. Release review must therefore score world-context depth separately from isolation.


## World Alignment Need And Relevance Arbitration

`v0.7` must not decide world-alignment behavior through coarse book-type or
skill-type routing. A principle, philosophical model, business method, or financial
method all use the same arbitration model. The model first asks how much world
alignment is needed, then asks whether each candidate world pressure actually fits
the source-derived skill.

### Intervention Need Score

Each skill receives a `world_alignment_need_score` in the range `0.0` to `1.0`.
The score controls intervention strength; it does not by itself allow a world
pressure to enter the artifact.

Inputs:

- `temporal_need_score`: how much the skill depends on current conditions, dates,
  market data, regulations, tooling, organization state, or live context.
- `current_fact_dependency_score`: whether safe use requires facts not present in
  the source bundle.
- `application_risk_score`: how much harm or decision cost can result from direct
  use.
- `misuse_risk_score`: how likely users are to turn the source skill into summary,
  fact lookup, direct advice, or boundary-overreach.
- `user_actionability_score`: whether the skill is likely to drive concrete action
  rather than reflection only.

Intervention levels:

| Score | Level | Allowed behavior |
| ---: | --- | --- |
| `0.00-0.25` | `minimal` | Keep the source view primary; add only misuse boundaries, no-web disclosure, and optional context reminders. |
| `0.25-0.50` | `light` | Add application boundaries and user-context questions; do not expand the source view. |
| `0.50-0.75` | `moderate` | Add relevant world pressures, caveats, and concrete user-context checks. |
| `0.75-1.00` | `strong_gate` | In no-web mode, require context or refuse direct application when current facts are needed. |

### Candidate Relevance Arbitration

Every candidate world pressure must pass relevance arbitration before it can appear
in `world_context.yaml` or `WORLD_ALIGNMENT.md`. Structural richness alone is not
enough.

Candidate dimensions:

- `source_fit_score`: whether the pressure directly helps apply the source-derived
  judgment rather than merely sharing surface words. This is a hard gate.
- `enrichment_value_score`: whether the pressure makes application more useful by
  naming constraints, boundary conditions, or failure modes.
- `application_need_score`: whether the user would be materially less safe or less
  effective without this pressure.
- `dilution_risk_score`: whether the pressure pulls the skill away from the source
  idea, over-modernizes it, or replaces reflection with fashionable context.
- `hallucination_risk_score`: whether the pressure contains unverified current
  facts, live-world claims, or broad claims that no-web mode cannot justify.

Acceptance rule:

```text
candidate accepted only if:
  source_fit_score >= source_fit_min
  dilution_risk_score <= dilution_risk_max
  hallucination_risk_score <= hallucination_risk_max
  and candidate_acceptance_score >= acceptance_min

candidate_acceptance_score =
  source_fit_score
  + enrichment_value_score
  + application_need_score
  - dilution_risk_score
  - hallucination_risk_score
```

No candidate may bypass `source_fit_score`. If a candidate is current-fact heavy in
no-web mode, it may only become a question, caveat, or refusal reason; it may not
be stated as verified world truth.

### No Forced Enhancement

If no candidate pressure passes arbitration, the correct output is
`no_forced_enhancement`, not a weak or off-domain world-context paragraph. In that
case the world layer should say that the source skill remains best used in
original-source-only mode, while preserving misuse boundaries and hallucination
warnings.

This is especially important for low-temporal principles, philosophy-like mental
models, and reflective judgment skills. They may still need misuse protection, but
they should not be made less faithful or less rich by forced modern-world framing.

### Review Requirement

Review must score not only depth and structure, but also fit. A detailed world
context with poor source fit or high dilution risk must score lower than a minimal
context that honestly preserves the source idea. The review output should expose
`world_alignment_need_score`, `intervention_level`, `source_fit_score`,
`dilution_risk_score`, `hallucination_risk_score`, accepted/rejected pressure
counts, and whether `no_forced_enhancement` was used.

## Architectural Principle

World alignment uses a four-layer thinking model but a three-file engineering surface.

Thinking model:

1. `source_truth`: source-faithful skill, anchors, claims, and native boundaries.
2. `world_context`: isolated real-world context and minimal time metadata.
3. `user_context`: the current user situation, constraints, and missing facts.
4. `application_gate`: the only decision outlet that says how, whether, or whether not to apply the skill.

Engineering surface for v0.7.0:

1. `world_context.yaml`
2. `application_gate.yaml`
3. `WORLD_ALIGNMENT.md`

`temporal_policy`, `pressure_tests`, `decision_path`, and `scorecard` remain conceptual concerns, but v0.7.0 should implement them through the three-file surface instead of creating a larger object graph.

## Proposed Bundle Surface

```text
bundle/
  skills/
    <skill_id>/
      SKILL.md
      anchors.yaml
      eval/
      iterations/
      usage/

  world_alignment/
    world_context.yaml

    <skill_id>/
      application_gate.yaml
      WORLD_ALIGNMENT.md

  traces/
    world_context/
      <case_id>.yaml
```

`skills/` remains source-faithful. `world_alignment/` is the only place where external reality, temporal caveats, pressure tests, and application-gate decisions may be stored.

A generated bundle must remain usable without `world_alignment/`. When world alignment is disabled, the system should expose the original source-faithful skill and its native boundaries rather than fail or silently fall back to blended advice.

## World Context Model

A world-context item keeps only the minimum time metadata needed for v0.7.0. Release-time and use-time contexts remain one schema, but live web retrieval is deferred.

Example:

```yaml
id: ai-prototyping-changed-requirements-work
context_type: release_baseline
observed_at: 2026-04-24
freshness_policy: on_release
temporal_sensitivity: medium
confidence: medium
source_summary: AI prototyping, low-code delivery, and compliance expectations can change how requirements-analysis methods are applied in modern organizations.
applies_to:
  - solution-to-problem-reframing
  - workflow-gateway
allowed_effects:
  - add_usage_caveat
  - add_pressure_test
  - require_current_context
forbidden_effects:
  - rewrite_source_rationale
  - replace_source_evidence
```

Use-time or runtime-compatible example:

```yaml
id: named-company-current-regulatory-status
context_type: user_supplied_or_external_check_required
observed_at: user_supplied_or_future_check_required
freshness_policy: before_use
temporal_sensitivity: high
required_when:
  - named_current_company
  - regulatory_or_compliance_claim
allowed_effects:
  - gate_application
  - ask_more_context
forbidden_effects:
  - rewrite_source_rationale
  - replace_source_evidence
```

## Minimal Time Strategy

Temporal alignment strength should be proportional to temporal sensitivity and application risk, but v0.7.0 intentionally keeps the time model small.

Required fields:

```yaml
source_period: string | null
observed_at: date | user_supplied_or_future_check_required
freshness_policy: none | on_release | before_use
temporal_sensitivity: low | medium | high
```

Default handling:

- `low`: original-source-only mode is usually sufficient; world alignment is optional and mostly checks misuse.
- `medium`: release-baseline context may add caveats or ask for user context before concrete application.
- `high`: do not pretend to know current facts; if before-use context is unavailable, return `ask_more_context` or `refuse`.

Deferred from v0.7.0:

- live web search
- automatic real-time fact verification
- `valid_until` lifecycle management
- multi-world / multi-stance modeling
- automatic investment or financial current-data checks

## Application Gate

The application gate is the final world-alignment decision surface. It does not rewrite the skill; it decides whether the skill can be applied under the available context.

Allowed verdicts:

```yaml
verdict: apply | apply_with_caveats | partial_apply | ask_more_context | refuse
source_skill_unchanged: true
reason: string
required_context: list[string]
decision_path: list[string]
```

The gate must preserve a visible path:

1. source skill and source anchors
2. relevant world context items
3. user-context or missing-context check
4. application verdict
5. caveats or refusal reason

## Required Scores

`v0.7` introduces world-alignment scoring without replacing existing source/generated/usage review.

Recommended dimensions for v0.7.0:

- `source_fidelity_score`
- `world_context_isolation_score`
- `application_gate_quality_score`
- `world_alignment_explanation_score`
- `workflow_boundary_resilience_score`
- `world_context_depth_score`

Initial release gate:

```yaml
v0.7_release_gate:
  source_fidelity_preserved: true
  world_context_isolated: true
  workflow_boundary_preserved: true
  samples_passed_min: 3
  world_alignment_score_min: 85
  world_context_depth_score_min: 80
  source_pollution_errors: 0
  live_web_required: false
  multi_world_modeling_required: false
  external_blind_review_required: false
  real_user_validation_required: false
  original_source_only_mode_supported: true
  application_gate_cases_min: 9
```

## Sample Scope

`v0.7` should be verified on at least three independent samples:

- Poor Charlie: low / medium temporal sensitivity, principle-heavy skills
- Effective Requirements: medium temporal sensitivity, method and organization-context skills
- Financial Statement: medium / high temporal sensitivity, concrete application and current-context gates

Each sample should prove:

- original `skills/` are not polluted by world context
- `world_alignment/` artifacts exist independently
- selected skills have `world_context.yaml`, `application_gate.yaml`, and `WORLD_ALIGNMENT.md`
- application gate can produce at least three verdict types across the sample set
- review separates source fidelity from world usefulness
- no sample requires live web search or multi-world modeling to pass v0.7.0

## Release Claim

A successful `v0.7` release may claim:

- KiU can keep source-derived skills stable while adding a separate world-alignment layer.
- KiU can explain when a skill is applicable, stale, risky, incomplete, or blocked by missing current context.
- KiU can improve internally verified practical usefulness without using external world context as hidden source evidence.

It may not claim:

- fully automated live-world factual correctness
- a complete runtime world-checking product
- replacement of human or domain-expert judgment in high-stakes domains
- human blind-review preference or real user validation
- that source-derived skills are rewritten to be universally current
