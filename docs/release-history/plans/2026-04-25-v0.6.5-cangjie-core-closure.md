# v0.6.5 Cangjie Core Closure Work Order Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. This plan follows the KiU `基 -> 标 -> 差 -> 策 -> 拆` method. Do not skip directly to implementation without verifying the baseline and target criteria.

**Goal:** Close the v0.6.5 cangjie core absorption gap with evidence that measures deep book-to-skill methodology, not only immediate usage quality.

**Architecture:** Keep KiU's default generation path raw-source-first. Add methodology evidence around the existing pipeline: baseline matrix, per-skill RIA-TV provenance, decoy pressure tests, clean same-source benchmark, and optional blind preference evidence. These artifacts feed split methodology scores (`internal`, `external_blind`, and `closure`) plus `final_artifact_effect` without weakening workflow-vs-agentic routing.

**Tech Stack:** Python standard library, PyYAML, unittest, existing KiU modules under `src/kiu_pipeline`, CLI scripts under `scripts/`, backlog tracking in `backlog/board.yaml`, reports under `reports/`, temporary generated artifacts under `/tmp`.

---

## AGENTS.md Alignment

This plan is constrained by four project-level principles:

- External reference boundary: `cangjie-skill` is reference/benchmark material, not a hidden upstream generator for default KiU artifacts.
- Workflow-vs-agentic boundary: high workflow certainty plus high context certainty remains `workflow_script_candidate` under `workflow_candidates/`, never a thick skill.
- System efficiency over local advantage: closure must be pipeline-level and repeatable, not manual editing of generated artifacts.
- Five-step landing method: define baseline before target, target before gap, gap before strategy, strategy before executable tasks.

| Gap ID | Missing Capability | Current Symptom | Required Fix |

| G8 | Score gate previously overclaimed | Usage wins could imply closure. | Keep cap until pressure/blind/same-source evidence exists. |

## 4. Strategy: How To Cross The Gap

Chosen strategy: close cangjie core through evidence-depth gates, not through output-count or usage-score inflation.

Implementation choices:

- Keep `v0.6.5` as the active closure version if it has not been released; do not open v0.6.6 only to move unfinished goals forward.
- Reuse existing work orders `KIU-687`, `KIU-688`, `KIU-693` through `KIU-699`; do not reuse `KIU-700` through `KIU-706` because those IDs already belong to v0.7 world-alignment.
- Treat pressure testing as mandatory for cangjie closure; treat blind preference as mandatory interface plus optional external evidence unless the user explicitly supplies human review inputs.
- Preserve workflow/agentic boundary even if this lowers output count or apparent cangjie similarity.
- Never modify `/tmp` generated artifacts manually; all quality gains must come from pipeline code, schemas, tests, or benchmark logic.

Rejected strategies:

- Do not raise `cangjie_methodology_closure_100` by changing weights only; internal and external evidence must remain separately visible.
- Do not ingest cangjie final `SKILL.md` outputs into KiU generation.
- Do not make every chapter or case into a skill to improve coverage numbers.
- Do not require a UI/review platform for v0.6.5; local file-based blind evidence is enough for this version.
