# v0.7.2 Application Readiness Model Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Improve final artifact quality by adding a general Application Readiness Model made of coverage, mechanism evidence, and transfer fit submodels.

**Architecture:** Add focused model modules and orchestration reports. Samples are used only as regression diagnostics; production logic must not depend on sample-specific names or paths.

**Tech Stack:** Python standard library, `unittest`, JSON/YAML evidence artifacts, existing KiU generation and review pipeline.

---

- Create: `src/kiu_pipeline/coverage_model.py` - source capability coverage model and reports.
- Create: `src/kiu_pipeline/mechanism_evidence.py` - mechanism density scoring and anchor-role decisions.
- Create: `src/kiu_pipeline/transfer_fit.py` - transfer fit dimensions and generated constraint questions.

- Create: `tests/test_application_readiness.py` - shared model tests.
- Create: `tests/test_coverage_model.py` - coverage model tests.
- Create: `tests/test_mechanism_evidence.py` - mechanism evidence tests.
- Create: `tests/test_transfer_fit.py` - transfer fit tests.

## KIU-748: Application Readiness Schema

**Purpose:** Establish the shared schema before adding submodels.

- [ ] Write failing tests for `ReadinessStatus`, `ReadinessFinding`, and aggregate readiness summary.
- [ ] Verify tests fail because `kiu_pipeline.readiness` does not exist.
- [ ] Implement minimal dataclasses/enums and `to_dict()` methods.
- [ ] Run `PYTHONPATH=src .venv/bin/python -m unittest tests.test_application_readiness -v` and confirm pass.
- [ ] Commit with message `Add application readiness schema`.

Acceptance:

- Status values include `pass`, `warn`, `fail`, and `not_applicable`.
- Findings carry `model`, `severity`, `reason`, `evidence`, and `recommended_action`.
- Aggregation never hides warnings behind a high numeric score.

## KIU-749: Coverage Model Orchestration

**Purpose:** Detect narrow output shape as a source-coverage issue, not a hard skill-count rule.

- [ ] Write failing tests with synthetic graph communities showing multiple high-density clusters and only one generated artifact.
- [ ] Add negative tests where one artifact is acceptable because coverage is explicitly justified.
- [ ] Implement `coverage_model.build_coverage_report()`.
- [ ] Orchestrate report writing to `reports/coverage-readiness.json`.
- [ ] Run coverage tests and affected pipeline tests.
- [ ] Commit with message `Add coverage readiness model`.

Acceptance:

- The model identifies `coverage_units`, `covered_units`, `uncovered_units`, and `narrow_output_justification`.
- It does not require a fixed number of skills.
- Workflow candidates count as coverage when routing evidence and gateway/audit trail exist.
- Effective Requirements may diagnose breadth through coverage readiness, but production logic contains no Effective-specific checks.

## KIU-750: Mechanism Evidence Model

**Purpose:** Select primary anchors based on mechanism support rather than information density or source form.

- [ ] Write failing tests for high-mechanism evidence with actor/action/constraint/consequence.
- [ ] Write failing tests for mechanism-weak evidence such as pure title/list/note/genealogy text.
- [ ] Write a non-regression test proving a title/list can still be supporting context if it is not used as primary mechanism proof.
- [ ] Implement `mechanism_evidence.score_mechanism_evidence()` and `decide_anchor_role()`.
- [ ] Integrate mechanism scores into anchor selection / distillation metadata.
- [ ] Run mechanism tests plus render/pipeline tests.
- [ ] Commit with message `Add mechanism evidence model`.

Acceptance:

- Primary anchors for judgment skills require sufficient mechanism density.
- Mechanism-weak evidence can remain in graph/source context but is downranked for primary rationale.
- No sample title, book name, or specific chapter path is hard-coded.
- Existing chapter-title hygiene is reinterpreted as a mechanism-evidence guardrail, not a standalone sample fix.

## KIU-751: Transfer Fit Model

**Purpose:** Generate concrete constraint questions for transfer use cases from mechanism dimensions.

- [ ] Write failing tests for transfer fit questions with mechanism fit, incentive mismatch, context mismatch, time horizon, resource/authority, disconfirming evidence, and anti-condition dimensions.
- [ ] Write non-dilution tests proving low-risk reflection does not receive heavy transfer questioning.
- [ ] Implement `transfer_fit.build_transfer_fit_report()`.
- [ ] Integrate transfer fit into application gate output for `transfer_candidate` use state.
- [ ] Run transfer fit, use-state, proxy, and world-alignment tests.
- [ ] Commit with message `Add transfer fit model`.

Acceptance:

- `transfer_candidate` gates include at least three specific constraint questions when transfer readiness is not complete.
- Questions are derived from mechanism dimensions, not fixed generic templates.
- `low_risk_reflection` remains minimal/light and is not forced into operational transfer checks.

## KIU-752: Cross-Sample Readiness Evaluation

**Purpose:** Prove the model changes generalize across the diagnostic samples without sample-specific patches.

- [ ] Rerun Financial Statement, Effective Requirements, Poor Charlie, Shiji, and Mao.
- [ ] Collect `coverage-readiness.json`, mechanism evidence reports, transfer fit reports, world alignment, proxy usage, and three-layer review for each sample.
- [ ] Update or create a final readiness evidence report under `reports/`.
- [ ] Run staged tests: focused model tests, v0.7 regression subset, then full unittest suite.
- [ ] Commit with message `Add application readiness evaluation evidence`.

Acceptance:

- Five-sample proxy average remains >= 90.
- No release proxy evidence has `wrong_verdict` or `boundary_leak`.
- Internal final artifact value average reaches >= 92.
- Effective Requirements-style narrow output is either improved by general coverage orchestration or explicitly justified by coverage readiness.
- Shiji-style mechanism-weak primary anchor issue is addressed by mechanism evidence scoring, not sample-specific filtering.
- Source pollution remains 0.
- External blind and cangjie victory remain unclaimed.
