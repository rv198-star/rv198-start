# KiU v0.6 规划意见与工单计划（基于 v0.5 复核）

> Status note (`2026-04-24`): this document remains the archive for the `v0.6`
> Graphify/upstream-graph line. `In Use world-alignment` is no longer considered part
> of `v0.6`; it is reserved for the separate `v0.7` line. See
> [`docs/2026-04-24-version-roadmap-v051-v06-v07.md`](./2026-04-24-version-roadmap-v051-v06-v07.md).

## 1. 结论摘要

基于对已发布 `kiu-skills-v0.5.0` 的复核，`v0.6` 的方向不应该是“重新定义 foundation”，而应该是：

1. 以 `v0.5` 为稳定底座，不重写已经成立的能力。
2. 把真正缺失的能力收敛为一条主线：`raw book -> extraction -> graph -> distillation -> routing -> review`。
3. 用端到端产出质量，而不是 schema 完整度，作为 `v0.6` 的第一性目标。

### 1.1 `v0.6.0` 版本目标冻结

`v0.6.0` 的唯一主目标，不是继续主攻 skill 文风润色，也不是提前引入
`world-alignment`，而是把 KiU 的上游素材消费链补成可审计、可交接、可验证的
生产段。

正式口径：

> `raw book -> source chunks -> extraction results -> provenance-rich graph -> candidate distillation -> workflow/agentic routing -> three-layer review`

### 1.2 `Graphify core only`

`v0.6.0` 只吸收 `Graphify` 的核心能力，不吸收外围包装。

纳入范围：

- provenance-rich graph schema
- `EXTRACTED / INFERRED / AMBIGUOUS` 三态标记
- `deterministic pass -> LLM pass` 两段式提取纪律
- graph/source provenance 与 extraction audit 合同

不纳入 `v0.6.0` 主目标：

- 多 IDE 集成
- 多语言 tree-sitter 扩展面
- MCP / serve / watch / export 包装
- 为了“图更好看”而提前做重展示层工程

### 1.3 `In Use first`

`v0.6.0` 不是“过程更优美”的版本，而是“更能支撑 In Use 产物”的版本。

因此：

- 首先看上游素材链是否能稳定支撑后续 skill/workflow 生产
- 不以图的外显可视化质量作为首发版本门槛
- `clustering` 与 `GRAPH_REPORT.md` 在 `v0.6.0` 中属于强 `P1`，不是首发 `P0 release gate`

### 1.4 可外包级成熟度原则

`v0.6.0` 的上游生产链必须达到“可外包级”的工程成熟度，也就是足以达到企业
信息化 `L4+` 水平的可交接性。

它不等于“真的把业务外包”，而是要求：

- 关键中间资产不能只存在于某次 LLM 对话里
- 更换模型、操作者、执行轮次后，核心证据骨架仍然稳定、可审计、可移交
- LLM 可以是必要生产力，但不能是唯一记忆体、唯一状态容器、唯一证据来源

### 1.5 证据骨架与 LLM 合成的分工

`v0.6.0` 不要求“没有 LLM 也能完整产出 skills”，但要求：

- `deterministic baseline for evidence spine`
- `LLM-required synthesis for skill production`

即：

- deterministic 层负责 `source chunks`、`source location`、schema validity、
  provenance binding、基础 extraction audit
- LLM 层负责 richer extraction、relation disambiguation、candidate distillation、
  drafting、review/refinement

这条原则的目的，是防止上游关键资产退化为不可审计的黑箱对话产物。


### 1.6 底盘闭合不等于终局 usage 大幅胜出

`v0.6.0` 的主目标是吸收 Graphify core，闭合素材消费、证据坐标、三态图和图导航底盘。
它不应被叙述成“最终 SKILLS 使用体验已经大幅碾压 cangjie-skill”。

更准确的判断是：

- `v0.6.0` 要证明 KiU 拥有更强的长期证据底盘和可审计性。
- same-source usage 需要保持领先，但不要求在本版本出现数量级优势。
- 大幅拉开最终使用效果，要靠后续 graph-to-skill distillation 和 world-alignment。

## 2. 对 v0.5 审计的修正意见

### 2.1 关于 pipeline

`v0.5` 不能被描述为“pipeline = 0”。

更准确的说法是：

- `v0.5` 已经有 **seed/refinement candidate pipeline**
- `v0.5` 缺的是 **raw-book extraction pipeline**

也就是说，`v0.6` 的任务不是“从零开始发明 pipeline”，而是补齐从原书到 source bundle / graph seeds 的上游生产段。

### 2.2 关于 Agentic / Workflow

审计里对这条线的评价方向基本对，但低估了 `v0.5` 的完成度。

`v0.5` 已经不只是：

- `routing_rules`
- `workflow_candidates/`
- 评分时计入 `workflow_boundary_factor`

它还会实际生成：

- `workflow.yaml`
- `CHECKLIST.md`

所以 `v0.6` 不该重做“边界原则”，而应该做两件更具体的事：

- 让 `workflow_certainty` 的判定尽量从抽取证据中长出来，而不是长期依赖人工 override
- 增加 workflow 候选的数量与分布，证明边界机制可规模化

### 2.3 关于 Graphify / cangjie 对标

这两条参照线可以继续用，但口径要更严格：

- `Graphify 25%`、`cangjie 45%` 这类百分比只能当内部判断，不应当成“可验证事实”
- `5 vs 12` 可以作为 benchmark 目标，但 `12` 这半句如果没有在本 repo 内部复验，不应当写成既成事实
- `5 vs 12` 只能说明 throughput / granularity 差异，**不能直接推出质量优劣**

`v0.6` 计划里建议改成：

- `Graphify-style provenance graph` 是目标形态
- `cangjie-style book-to-skill throughput` 是 benchmark
- 不再写不可审计的完成度百分比

### 2.4 关于 benchmark 口径

`v0.6` 的 benchmark 必须从 `count-first` 调整为 `quality-first`。

也就是说，后续评估顺序应变成：

1. `数量 / 覆盖`
   - 只作为外层概览
   - 用来暴露 throughput 与 granularity 差距
2. `概念对齐后的产物质量`
   - 对 same-source reference pack 做 `concept alignment`
   - 比较 trigger / boundary / actionability / evidence traceability
3. `同场景使用效果`
   - 用同一批 scenario 做 usage-level 对照
   - 比较误触发、空话率、下一步动作质量、边界纪律

因此，后续 release gate 里，“skill 数量”只能是辅助指标，不能单独作为版本质量结论。

## 3. v0.6 的规划意见

## 3.1 不要重写的东西

`v0.5` 里以下能力已经成立，应视为回归基线：

- 多 domain bundle 结构
- 跨 bundle graph merge
- Agentic / Workflow 路由
- 三层审查评分
- production quality gate

`v0.6` 开发时，这些模块默认只允许“向前兼容扩展”，不做推倒重写。

## 3.2 v0.6 的真实主目标

`v0.6` 的核心目标应定义为：

> 从单本原书文本出发，稳定产出带出处信息的 graph、可筛选的 skill/workflow 候选，以及可复核的三层审查结果。

这意味着 `v0.6` 的优先级顺序应是：

1. 原书 ingestion / chunking
2. extraction outputs 的中间表示
3. provenance-rich graph schema
4. distillation / routing / review 接线
5. graph report / clustering / 辅助可视化

补充一条 `v0.6.0` 顺手解决的项目基础设施项：

- 增加仓库内原生 `backlog/kanban` 资产，记录跨版本工单状态、阻塞级别、验收标准与证据链接，避免后续 AI 执行只依赖本地 `plan` 文本而丢失项目级连续性
- canonical 入口固定为 `backlog/board.yaml` 与 `python3 scripts/show_backlog.py`

也就是说，Leiden 聚类和 `GRAPH_REPORT.md` 是有价值的，但不应抢在 extraction IO 合同之前成为 W1 主目标。

## 3.3 v0.6 的发布门槛建议

建议把 `v0.6.0` 的验收门槛改成更可审计、且明确多样本的版本：

- 仓库内至少维护 `3` 本独立样本书
- 至少 `2` 本作为正式 production-validation 样本，要求独立端到端跑通
- `Poor Charlie's Almanack` 也作为正式样本，但角色偏 `benchmark-control`
- 三本样本必须独立运行，不允许混成一个复合案例
- graph 中每个 candidate 都有 provenance 字段
- 上游关键资产必须满足“可外包级成熟度”：`source chunks / extraction results / graph / verification logs`
  都可落盘、可审计、可移交
- 端到端结果可落 `reports/three-layer-review.json`
- same-source benchmark 至少完成一轮 `concept alignment + artifact review + same-scenario usage review`
- 对 `Poor Charlie` 的最终判断采用“分层超越”标准：
  - `usage / boundary / evidence honesty` 必须明显领先 `cangjie`
  - 数量不要求硬超，但不能明显落后，且 skill/workflow 拓扑必须更合理

相比“必须 15+”或“数量必须双超”，这个门槛更适合第一版上游生产管线，因为它既保留了压力，也避免为了凑数量牺牲质量。

## 3.4 v0.6 的外部参照评分卡

后续版本评估不应只看“KiU 自己有没有进步”，还要看是否真正吸收了两个参照项目的核心。

建议从 `v0.6` 开始固定一张内部评分卡，作为阶段评审模板：

### Graphify 核心参照

只看内核，不看外围：

- `graph.json schema`
  - 是否具备 `source_file`
  - 是否具备 `source_location`
  - 是否具备边级 `confidence`
  - 是否具备边级 `extraction_kind`
- 三态边标记
  - 是否明确区分 `EXTRACTED / INFERRED / AMBIGUOUS`
- 两段式提取架构
  - 是否有 deterministic pass
  - 是否有 LLM pass
  - 是否能保留审计轨迹
- 社区检测
  - 是否从图自动生成 communities，而不是长期手工维护
- `GRAPH_REPORT.md`
  - 是否能输出 god nodes / communities / surprising connections / suggested questions

注：

- `clustering` 与 `GRAPH_REPORT.md` 在 `v0.6.0` 里是强 `P1`
- 它们是有价值的 Graphify core 能力，但不应抢在上游 extraction/provenance 合同之前成为首发门槛

### cangjie-skill 核心参照

只看方法论与生产内核：

- `RIA-TV++` 风格的分阶段生产设计
- 五类 extractor 的明确分工
  - framework
  - principle
  - case
  - counter-example
  - term
- 三重验证规则
  - 跨域佐证
  - 预测力
  - 独特性
- 元 skill / book-to-skill 执行规范
- 模板文件体系
  - `SKILL.md`
  - `INDEX.md`
  - `BOOK_OVERVIEW.md`
  - `test-prompts.json`

### 建议评分方式

后续每个版本都按三组分开打：

1. `KiU foundation retained`
   - merge / routing / review / quality gate 是否保持
2. `Graphify core absorbed`
   - provenance graph / 三态边 / 两段式提取 / clustering / graph report
3. `cangjie core absorbed`
   - 分阶段 pipeline / extractor 设计 / 三重验证 / book-to-skill 模板

这样做的好处是：

- 不会把外围生态误算成“核心能力”
- 不会把不可验证的主观百分比当作事实
- 可以直接看到每个版本究竟是在补 graph 核心，还是在补蒸馏核心

## 3.5 v0.6 benchmark 分层

后续 benchmark 固定分三层：

### Layer 1 · Pack Overview

- `output_count`
- `coverage`
- `workflow_vs_agentic_boundary`

### Layer 2 · Concept-Aligned Artifact Review

- `concept_alignment`
- 对齐 pair 的 `kiu_review`
- 对齐 pair 的 `reference_review`
- `artifact_score_delta_100`

### Layer 3 · Same-Scenario Usage Review

- 同一 scenario 下的使用结果对照
- 重点看：
  - trigger 命中是否更准
  - boundary 是否更稳
  - 输出是否更具体
  - 下一步动作是否更可执行

这三层里，`Layer 2` 和 `Layer 3` 是质量主线，`Layer 1` 只做外层概览。

## 4. 建议的四周拆解

### 启动前清尾项

在正式进入 `KIU-610` 之前，先清一个 `v0.5.1` carry-over 小尾项：

- `KIU-516`：narrow valuation residual cleanup

原因：

- `v0.5.1` 虽然已经满足 release gate 并正式发布，但 scorecard 仍记录了一个窄残差：
  `value-assessment-source-note` 相对 reference `value-assessment` 还落后 `-1.7`
- 这不是阻断版本发布的问题，但它是最明确、最小、最值得先收掉的 pair-level 质量残留
- 先收这一步，再冻结 `v0.5` baseline，更能避免后续把 `v0.5.1` 的未尽质量残差混进 `v0.6.0` 主线

因此 `v0.6.0` 的实际开工顺序应为：

1. `KIU-516` narrow valuation residual cleanup
2. `KIU-610` v0.5 baseline freeze
3. `KIU-620` book overview / extraction context contract
4. `KIU-613` graph schema v0.2
5. `KIU-622` triple verification gate

### W1：基线冻结 + provenance 合同

目标：把 `v0.5` 作为不可退化底座固定下来，同时定义 `v0.6` 的 source/extraction/graph 合同。

交付：

- attribution 文件就位
- `v0.5` 回归基线命令固定
- `graph schema v0.2` 定稿
- source chunk / extraction output schema 定稿
- 当前两个 bundle 的 graph 迁移脚本可跑

### W2：提取前半段

目标：从原书文本走到“可用的 extraction outputs”。

交付：

- source ingestion
- chunking / section map
- deterministic extraction skeleton
- LLM extraction interface
- provenance-rich interim artifacts

### W3：蒸馏与分流

目标：把 extraction outputs 接到 KiU 现有 candidate / routing / review 管线上。

交付：

- extracted graph -> candidate seeds
- candidate 去重 / 过滤 / ranking
- Agentic / Workflow 自动分流
- workflow 候选数量从 `1` 提升到 `>=2`

### W4：端到端 benchmark

目标：拿《穷查理宝典》做第一轮真实生产 benchmark。

交付：

- 完整 graph
- 10-15 个 `good+` candidate skills
- 2 个以上 workflow candidates
- 三层审查报告
- 对照 benchmark 的质量总结
- same-source `concept alignment` 结果
- 至少一轮 same-scenario usage benchmark

## 5. W1 工单计划（建议版）

### KIU-610 · 冻结 v0.5 回归基线

目标：

- 明确哪些能力在 `v0.6` 绝不能退化

## 6. 对 W1 排序的建议

建议顺序：

1. `KIU-516` valuation residual cleanup
2. `KIU-611` Attribution
3. `KIU-610` v0.5 baseline freeze
4. `KIU-612` source ingestion schema
5. `KIU-613` graph schema v0.2
6. `KIU-614` graph migration
7. `KIU-615` extraction skeleton CLI
8. `KIU-616` examples regression set

原因：

- 先把法律和基线锁住
- 再定义输入输出合同
- 再迁移现有图
- 最后才开始搭 extraction 入口

这比“先做 Leiden / GRAPH_REPORT”更贴近 `v0.6` 的真实主矛盾。

## 7. 我的总判断

`v0.6` 应该被定义为：

> **在 `v0.5` 已成立 foundation 上，自研 raw-book production pipeline 的第一版。**

如果按这个定义推进，计划会更稳，也更符合这次复核后的事实。
