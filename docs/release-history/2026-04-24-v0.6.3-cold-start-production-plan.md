# KiU v0.6.3 Cold-Start Production Gate Plan

Date: `2026-04-24`

## 背景

v0.6.2 审计指出一个必须在 v0.6.x 内闭合的问题：此前 Poor Charlie 同源对比主要是从已有 KiU source bundle 重新生成 candidates，再与 cangjie reference pack 对比；这只能称为 `source_bundle_regeneration`，不能称为 `raw_book_no_seed_cold_start`。

这不是 cangjie 作为隐性输入的问题：cangjie 仍只作为 reference / benchmark 使用。但 release evidence 必须清楚区分“已有 bundle 再生成”和“原始书稿零 seed 冷启动”。

## v0.6.3 目标

v0.6.3 先做防误报和证据边界收口，再开展真正冷启动实验。

目标：

- 每个 generated run 都写出 `reports/pipeline-provenance.json`。
- `run_book_pipeline.py --input <raw.md>` 产物可被标记为 `raw_book_no_seed_cold_start`，前提是 scaffold 出来的 source bundle skills 数为 0。
- `build_candidates.py --source-bundle <existing>` 产物必须标记为 `source_bundle_regeneration`，不得被 benchmark 或 release note 误称为冷启动。
- reference benchmark scorecard 必须显式报告 `book_to_skill_cold_start_proven`。
- v0.7 world-alignment 开发必须等待 v0.6.x 的冷启动生产门禁闭合。

## 术语定义

### source_bundle_regeneration

输入是已有 KiU source bundle，例如：

```bash
scripts/build_candidates.py --source-bundle bundles/poor-charlies-almanack-v0.1
```

该模式可以验证 graph/schema/distillation/review 能力，但不能证明 raw-book 冷启动能力。即使 cangjie 没有进入生成链路，也不能称为“从 0 开始”。

### raw_book_no_seed_cold_start

输入是原始 markdown 书稿，例如：

```bash
scripts/run_book_pipeline.py --input examples/sources/effective-requirements-analysis-source.md
```

该模式必须满足：

- `input_kind = raw_markdown_book`
- `source_bundle_skill_count = 0`
- `uses_existing_source_skills = false`
- `uses_reference_pack_as_generation_input = false`
- `manual_generated_artifact_patch_allowed = false`

## 工单

### KIU-672 · Cold-start claim provenance and anti-misreport gate

为两条入口写入统一 provenance，并让 review / benchmark 读取它。

验收：

- raw-book pipeline 输出 `pipeline_mode = raw_book_no_seed_cold_start`。
- source-bundle builder 输出 `pipeline_mode = source_bundle_regeneration`。
- `three-layer-review.json` 和 reference benchmark 都暴露 cold-start 字段。

### KIU-673 · No-seed raw-book cold-start experiment design

设计真正冷启动实验：选择至少一本 KiU 未预置 seed skill 的书，从 raw markdown 走完整 pipeline，并与 cangjie same-source reference 分开对比。

验收：

- 实验命令固定。
- 输入源、reference pack、generated run 三者路径分离。
- 报告明确不得使用 external final `SKILL.md` 作为生成输入。

### KIU-674 · cangjie production-methodology absorption gap

把审计指出的 cangjie RIA-TV++、五类 extractor、三重验证、诱饵压力测试缺口纳入 v0.6.x 后续计划，但不得与 Graphify v0.6 的 provenance gate 混淆。

验收：

- 后续工单拆分 extractor responsibilities、triple verification、decoy pressure benchmark。
- benchmark 报告不再把 stage presence 等同于 production methodology fully absorbed。

### KIU-675 · Cold-start evidence pack and comparison closure

冷启动跑通后，产出新的 evidence pack，重新解释 Poor Charlie 既有同源比较的边界，并补充 true cold-start 对照。

验收：

- evidence pack 同时列出 `source_bundle_regeneration` 与 `raw_book_no_seed_cold_start` 结果。
- 如果 cold-start 质量未超过 cangjie，报告必须诚实记录缺口，不得用旧 seed bundle 结果替代。

## AGENTS.md 对齐

- External reference boundary：cangjie / Graphify 只能是 reference 或 benchmark，不可成为默认生成链路的隐性 upstream。
- Workflow-vs-Agentic boundary：为了追求冷启动输出数量，不能把 high/high workflow candidate 强塞进 thick skill。
- System efficiency over local advantage：最终判断看平均可重复生产质量，不看单个手工优秀样例。

## v0.7 前置条件

v0.7 world-alignment 可以继续讨论，但进入实现前必须先有 v0.6.x cold-start gate 证据。否则 world-alignment 会叠在一个尚未证实的 book-to-skill 生产链路上。

## Shiji Cold-Start Finding

`examples/shiji_md/resource` is now the first stress sample for multi-file raw-book no-seed cold start. It confirmed two things:

- KiU needed a Graphify-core style multi-file source coordinate path; this is now covered by `KIU-676`.
- After repeated author-heading boilerplate is filtered by `KIU-677`, `KIU-678` adds a source-level `narrative_pattern_signal` extractor so narrative history can produce `historical-case-consequence-judgment` without manually editing generated outputs.

Evidence: `reports/2026-04-24-v0.6.3-shiji-cold-start-smoke.md`.

## Same-Source cangjie Comparison Boundary

The Shiji cold-start smoke must not be described as a cangjie comparison by itself. KiU-only scores prove the KiU pipeline state, not relative quality.

Current local cangjie facts:

- `.references/cangjie-skill` contains `SKILL.md`, methodology documents, extractor prompts, and templates.
- It does not contain a Python/Node/Makefile/CLI runtime that can be executed as an official generator.
- No local same-source Shiji cangjie skill pack exists.

v0.6.3 therefore introduces an explicit benchmark-only adapter: `cangjie-skill/RIA-TV++-deterministic-adapter`. It uses the same raw markdown book and cangjie output shape, but is not an official cangjie run.

This adapter is acceptable only under the external reference boundary:

- It is isolated from KiU's default generation path.
- It writes `metadata.json` with `official_cangjie_run: false` and `benchmark_only: true`.
- It must not be used as hidden upstream input to KiU generated artifacts.
- Reports must call it `cangjie_protocol` or `RIA-TV++ adapter baseline`, not “official cangjie output”.

The first Shiji same-source result is unfavorable to KiU on usage quality: KiU wins artifact structure on the matched skill, but loses same-scenario usage by `8.1` points and misses one role-boundary reference skill. That gap is now part of the v0.6.x workstream, not a v0.7 world-alignment issue.
