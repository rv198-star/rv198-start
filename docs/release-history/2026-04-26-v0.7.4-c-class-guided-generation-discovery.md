# v0.7.4 C-Class Guided Generation Discovery

v0.7.4 should use the v0.7.3 Class C action-value taxonomy to improve generation quality. This document is a discovery and problem-definition pass, not an implementation patch.

Method: recursive five-step method from `AGENTS.md`.

```text
L1 Discover: world -> signal
L2 Define: signal -> problem
L3 Resolve: problem -> action
Feedback: execution -> recalibration
```

The goal is to reverse-diagnose the whole book-to-skill construction chain across five books before deciding what to change.

## L1 Discover: World -> Signal

### Baseline

v0.7.3 introduced a five-book C-class scorecard over v0.7.2 readiness outputs:

| Book | Traditional Overall | Proxy | C-Class Score | Signal |
| --- | ---: | ---: | ---: | --- |
| `Poor Charlie's Almanack` | `97.9` | `94.7` | `90` | Traditional and C-class scores agree. |
| `毛泽东选集 / Anthology` | `94.6` | `93.9` | `89` | Strong generated judgment skills, near 90. |
| `史记` | `95.5` | `93.9` | `88` | Strong individual skills, book coverage bounded by broad corpus. |
| `有效需求分析（第2版）` | `94.4` | `94.5` | `88` | Good skill/workflow split, workflow action value not yet scored. |
| `财务报表分析` | `92.6` | `94.9` | `72` | Traditional scores high, C-class score low. |

The key signal is not simply that `财务报表分析` is weak. The key signal is that the existing construction and evaluation chain can produce high traditional scores while publishing artifacts that do not look like strong action judgment skills.

### Target

Discover where the book-to-skill pipeline loses action-value identity.

The discovery target is a reproducible signal, not a fix:

```text
Which construction-stage signals predict low C-class action value before a candidate becomes a published SKILL.md?
```

### Gap

Current evidence shows different books stress different parts of the pipeline:

- `Poor Charlie` starts from mature source-bundle skills and does not stress raw-book container routing.
- `Mao` and `Shiji` stress borrowed-value historical/argumentative transfer and boundary control.
- `Effective Requirements` stresses workflow-vs-agentic routing.
- `Financial Statement` stresses textbook containers: concepts, exercises, mini cases, toolboxes, and application sections.

The pipeline currently sees these as separate surface cases. It lacks a common pre-publication signal for whether a candidate is an action skill, a workflow, a case, an exercise, a concept note, or source context.

### Strategy

Trace the common chain across all five books:

```text
source chunks
-> extraction graph
-> candidate seed mining
-> candidate kind routing
-> seed verification
-> pseudo-skill hygiene
-> refinement / production quality
-> rendered SKILL.md
-> usage/proxy/world review
-> C-class action-value score
```

Use the five-book scorecard to compare working and failing patterns.

### Breakdown: Observed Signals

#### Signal 1: Container candidates can pass final-decision as excellent

In `Financial Statement`, the following published skills are marked `ready_for_review` and `excellent` in `final-decision.json`:

```text
关键概念
练习题
自主练习
分析师工具箱
基本练习
应用分析
迷你案例
workflow-gateway
```

Several of these are container-shaped rather than judgment-contract-shaped. They can still receive high production quality because later stages can fill in contract, rationale, evidence, usage, and evaluation structure.

#### Signal 2: Chapter-title hygiene works but is too narrow

`Financial Statement` rejects `289` chapter-title candidates, `Mao` rejects `188`, and `Effective Requirements` rejects `14`. That means existing hygiene catches obvious title forms.

But it does not catch semantic containers such as:

```text
关键概念
练习题
自主练习
迷你案例
分析师工具箱
应用分析
```

These are not necessarily chapter-title patterns, but they are often not publishable action skills.

#### Signal 3: Usage/proxy can become post-hoc self-consistency

Once a container-shaped candidate is published, usage/proxy review can generate scenarios around that candidate and score trigger/boundary behavior. This can make a weak candidate look coherent after the fact.

This does not mean usage/proxy is useless. It means usage/proxy must not be the first gate that decides whether a candidate deserves to exist.

#### Signal 4: Effective Requirements shows the desired split

`Effective Requirements` publishes:

```text
solution-to-problem-reframing
stakeholder-resistance-tradeoff
workflow-gateway
```

and routes `12` workflow candidates. This is closer to the desired model: agentic skills for judgment-heavy cases, workflow candidates for deterministic procedures, gateway for routing.

#### Signal 5: Poor Charlie and Shiji avoid the issue because their candidates are already judgment-shaped

`Poor Charlie` and `Shiji` do not prove the container issue is absent. They simply do not stress it as strongly. Their candidate ids are already close to judgment contracts:

```text
circle-of-competence
bias-self-audit
historical-analogy-transfer-gate
historical-case-consequence-judgment
```

These are naturally action-skill-like.

### L1 Discovery Conclusion

The discovered signal is:

```text
The pipeline has strong structure/evidence gates, but it lacks a pre-publication action-skill identity signal. When a source book contains dense containers such as exercises, concepts, cases, or toolboxes, those containers can be promoted to published skills and then made structurally coherent by later stages.
```

## L2 Define: Signal -> Problem

### Baseline

Current gates answer many useful questions:

- Is there source evidence?
- Is the graph signal extracted/inferred/ambiguous?
- Does the candidate have boundary and usage review?
- Does it satisfy production quality?
- Does it avoid source pollution?
- Does it route obvious workflows?
- Does it reject obvious chapter titles?

But current gates do not reliably answer:

```text
Is this candidate's identity actually a publishable action skill?
```

### Target

Define a root problem that is general across all five books and measurable before implementation.

### Gap

The current system conflates three different qualities:

- `structural completeness`: the artifact has contract, rationale, evidence, usage, eval, revision.
- `source support`: the artifact is anchored in source graph or source text.
- `action-skill identity`: the artifact represents a usable judgment/action capability rather than a source container.

The first two are necessary, but they are not sufficient for Class C action value.

### Strategy

Define the problem at the candidate-promotion level, not at the book-specific output level.

### Problem Statement

When source graph nodes or communities correspond to semantic containers such as concepts, exercises, cases, toolboxes, application sections, or chapter-like materials, the candidate pipeline can promote them into `bundle/skills/` if they have enough source support and can be structurally refined. Because the pipeline lacks an explicit action-skill identity gate, later quality and usage evaluation may validate the structure around a weak identity rather than rejecting or rerouting it. This produces high traditional scores but low book-to-skill action value.

### What This Problem Is Not

- It is not a Financial Statement-specific issue.
- It is not solved by manually renaming or editing generated `SKILL.md` files.
- It is not solved by banning all Chinese titles or all exercise-like words.
- It is not a failure of source fidelity by itself.
- It is not a failure of world alignment or live fact verification.

### Falsifiable Indicators

v0.7.4 should be able to measure:

```text
container_candidate_leak_count
publishable_action_skill_ratio
action_skill_identity_score
book_to_skill_action_coverage_100
skill_action_value_average_100
workflow_action_value_status
candidate_route_distribution
```

Definitions:

- `container_candidate_leak_count`: number of published `SKILL.md` artifacts whose identity is primarily a source container rather than a judgment/action capability.
- `publishable_action_skill_ratio`: publishable action skills divided by all published non-gateway skills.
- `action_skill_identity_score`: candidate-level score for having primary action layer, judgment trigger, required user context, action-value output, boundaries, and feedback/disconfirmation role.
- `candidate_route_distribution`: counts of candidates routed to `bundle/skills/`, `workflow_candidates/`, `evaluation_material`, `case_library`, `concept_note`, `source_context`, or rejected.

### L2 Definition Conclusion

Root problem:

```text
KiU needs an explicit action-skill identity and routing model between candidate mining and publication. Without it, structurally valid source containers can become published skills, and downstream quality evaluation may become post-hoc self-consistency.
```

## L3 Resolve: Problem -> Action

### Baseline

v0.7.3 created the evaluation vocabulary but did not change generation behavior. v0.7.4 should use the vocabulary to improve the construction chain.

### Target

v0.7.4 should improve C-class scores through a general candidate-promotion model, not sample-specific patches.

Target scores:

| Book | Current C | v0.7.4 Gate | v0.7.4 Stretch |
| --- | ---: | ---: | ---: |
| `Financial Statement` | `72` | `88+` | `90+` |
| `Effective Requirements` | `88` | `90+` | `92+` |
| `Mao` | `89` | `90+` | `92+` |
| `Poor Charlie` | `90` | `90+` | `92+` |
| `Shiji` | `88` | `90+` | `92+` |

System gates:

```text
container_candidate_leak_count = 0 for published non-gateway skills
publishable_action_skill_ratio >= 0.90
all published non-gateway skills have action_skill_identity_score >= 0.85
candidate_route_distribution is reported for each five-book run
```

### Gap

The implementation needs a new model boundary, not a pile of title rules.

Missing pieces:

- A book-level action-value profile before candidate promotion.
- A candidate-level action-skill identity assessment.
- A routing target set for non-skill but useful source material.
- A release review that reports action identity before usage/proxy self-consistency.

### Strategy

Implement v0.7.4 in three batches.

#### Batch 1: Discovery Instrumentation

Add structured reporting without changing publication behavior first.

Planned tickets:

- `KIU-810`: candidate route distribution report.
- `KIU-811`: action-skill identity diagnostic score.
- `KIU-812`: five-book action identity baseline report.

Purpose: prove the diagnostic catches the five-book pattern before changing routing.

#### Batch 2: Candidate Routing Model

Add a general routing model after diagnostics are visible.

Planned tickets:

- `KIU-813`: action-skill identity gate before `bundle/skills/` publication.
- `KIU-814`: route semantic containers to `concept_note`, `case_library`, `evaluation_material`, `source_context`, or `workflow_candidates/`.
- `KIU-815`: preserve gateway behavior for workflow-heavy books.

Purpose: fix root cause at candidate promotion, not in rendered outputs.

#### Batch 3: Five-Book Regression And C-Class Review

Rerun all five books and compare old/new C-class scores.

Planned tickets:

- `KIU-816`: rerun five-book generation.
- `KIU-817`: produce updated five-book C-class scorecard.
- `KIU-818`: verify no regression in source/fact safety, workflow-vs-agentic boundary, and traditional release gates.

### Breakdown To Executable Actions

The first executable work should not change routing. It should add diagnostics and tests that fail on the known pattern:

1. Create fixtures for action-skill identity assessment.
2. Add `action_skill_identity_score` as a pure model function.
3. Add report output for `candidate_route_distribution` and `container_candidate_leak_count`.
4. Run diagnostics over the five existing v0.7.2 readiness outputs.
5. Only after the diagnostic reproduces the pattern, add routing behavior.

Positive fixtures:

```text
solution-to-problem-reframing
circle-of-competence
historical-analogy-transfer-gate
```

Negative fixtures:

```text
练习题
自主练习
关键概念
迷你案例
```

### Feedback Loop

After v0.7.4 implementation, feed results back into:

- L1 if a new class of weak candidate appears.
- L2 if the action-skill identity definition is too broad or too narrow.
- L3 if routing improves Financial Statement but regresses Effective Requirements, Mao, Poor Charlie, or Shiji.

## Final Discovery Decision

v0.7.4 should optimize the construction chain at the candidate identity and routing boundary.

The root cause is not low source quality alone and not a Financial Statement-specific title issue. The root cause is that KiU currently lacks an explicit action-skill identity model between source-supported candidate discovery and published skill generation.

The next step is to convert this discovery into an implementation plan with diagnostics first, routing behavior second, and five-book regression third.
