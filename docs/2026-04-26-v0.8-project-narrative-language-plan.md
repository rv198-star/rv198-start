# v0.8 Project Narrative and Unified Language Plan

v0.7.x has accumulated enough internal mechanisms that the project is becoming hard for first-time reviewers to understand. The issue is not only documentation volume. It is that many public-facing explanations still expose reference-project and release-history terminology such as Graphify, cangjie-skill, absorption, closure, proxy gates, and internal methodology names before explaining KiU's own architecture.

v0.8 should treat this as a project narrative problem, not a small glossary cleanup.

## Goal

Create a KiU-native explanation of `学以致用`: how source knowledge becomes usable judgment while preserving evidence, boundaries, and user choice.

The main explanatory path should use KiU concepts first. External projects remain credited, but they should not be required vocabulary for understanding the system.

## v0.8 Migration Order

v0.8 must not begin with an old-term translation table. The order is:

1. Define the new architecture language.
2. Review coverage against current project capabilities.
3. Review Chinese naming and first-time reader clarity.
4. Freeze old terms as historical/internal vocabulary.
5. Write the historical-to-v0.8 glossary.
6. Refresh README only after the architecture narrative and glossary exist.

This keeps v0.8 from becoming a superficial glossary rename.

## Proposed KiU-Native Language

The v0.8 public architecture should use action phrases first:

```text
读准原书 -> 提炼判断 -> 生成技能 / 分流流程 -> 校准应用 -> 验证价值
```

The noun-heavy layer names remain useful as precise internal aliases, but README
and first-time-reader docs should prefer the action phrases.

## Architecture Narrative Target

The main v0.8 narrative should explain the pipeline in one chain:

```text
Book -> Read Accurately -> Distill Judgment -> Skill or Workflow -> Calibrate Use -> Verify Value
```

The narrative should answer:

- What problem does KiU solve?
- Why is KiU not a summary engine, translator, RAG note system, or encyclopedia query layer?
- How does KiU preserve source fidelity while making knowledge usable?
- Why does KiU separate agentic skills from deterministic workflows?
- Why does world/live context stay isolated instead of rewriting source skills?
- How does KiU evaluate method integrity, source/fact safety, and action value separately?

## Required Outputs

- `docs/project-architecture-narrative.md`: first-time-reviewer explanation of the whole system.
- `docs/concept-language-glossary.md`: old/internal terms mapped to stable KiU-native concepts.
- `docs/user-facing-evaluation-guide.md`: user-readable score interpretation that projects engineering metrics into source trust, action helpfulness, boundary clarity, coverage fit, context application safety, and evidence confidence.
- README or `docs/usage-guide.md` entry update pointing to the narrative before release evidence.
- Attribution references preserved in `NOTICE` and `docs/third-party-attribution.md`.

## Project Optimization Directions Recorded After v0.7.4

The v0.7.4 retrospective identifies four likely next directions. v0.8 should start with the first one and only pull the others forward if they serve the narrative cleanup.

## Boundaries

- Do not erase external influence or attribution. Move it out of the main explanatory path, not out of project history.
- Do not rename historical release evidence retroactively.
- Do not use the simplification effort to weaken evidence honesty or workflow-vs-agentic boundaries.
- Do not introduce new generation behavior as part of the narrative cleanup.
