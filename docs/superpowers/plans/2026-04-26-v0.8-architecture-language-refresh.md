# v0.8 Architecture Language Design

Date: 2026-04-26

v0.8 rebuilds KiU's public language around `学以致用`: turning source knowledge into usable judgment while preserving source evidence, boundaries, and user choice.

This is a language and architecture clarity release. It does not rename historical evidence and does not change generation behavior by itself.

## Core Architecture

```text
原书 -> 原书证据层 -> 判断提炼层 -> 行动技能层 / 流程路由层 -> 应用校准层 -> 行动价值验收
```

| Layer | Public name | What it does | What it must not do |
| --- | --- | --- | --- |
| 1 | 原书证据层 / source evidence layer | Preserve source facts, passages, structure, claims, anchors, and provenance. | Do not pretend source evidence is already a user skill. |
| 2 | 判断提炼层 / judgment distillation layer | Extract transferable judgment from source evidence. | Do not reduce the book to summaries, quotes, or encyclopedia answers. |
| 3A | 行动技能层 / action skill layer | Publish skills that help users judge, choose, refuse, act, or recalibrate. | Do not publish source containers, chapter headings, exercises, or workflows as thick skills. |
| 3B | 流程路由层 / workflow routing layer | Keep deterministic procedures auditable as workflows and route users to them when appropriate. | Do not collapse judgment-heavy skills into checklists or inflate checklists into skills. |
| 4 | 应用校准层 / application calibration layer | Add isolated current context, fact checks, caveats, and safety gates when needed. | Do not rewrite source-faithful skills into world-blended artifacts. |
| 5 | 行动价值验收 / action-value evaluation | Check whether outputs help users discover signals, define problems, resolve actions, and calibrate feedback. | Do not claim real-user or external validation from internal evidence. |

## Dragon-Eye Phrases

`技能不是摘要` means KiU does not exist to summarize books. It turns source knowledge into bounded, triggerable judgment.

`用而不染` means application context may help users apply a skill, but it must not pollute or rewrite the source-derived skill.

## Coverage Review

| Current capability | v0.8 architecture home | Coverage judgment |
| --- | --- | --- |
| Source bundles, graphs, anchors, provenance | 原书证据层 | Covered. |
| Candidate extraction, distillation, verification stages | 判断提炼层 | Covered. |
| Published `bundle/skills/*/SKILL.md` | 行动技能层 | Covered. |
| Workflow candidates and workflow gateway | 流程路由层 | Covered. |
| World context, live fact verification, hallucination gates | 应用校准层 | Covered. |
| A/B/C metrics, three-layer review, user-facing scorecards | 行动价值验收 | Covered. |
| External benchmarks and blind packs | Evidence confidence inside 行动价值验收 | Covered as evidence tier, not core architecture. |

## Clarity Review

| Check | Result |
| --- | --- |
| Can a new reader explain the project in 90 seconds? | Yes: KiU turns books into usable judgment, not summaries. |
| Does the architecture avoid excessive new jargon? | Yes: five layer names and two memorable phrases. |
| Does it depend on knowing reference projects? | No: reference projects move to attribution and historical mapping. |
| Does it preserve engineering boundaries? | Yes: source fidelity, workflow routing, and application isolation remain explicit. |
| Can it drive README structure? | Yes: the README should present problem, architecture, evidence, then usage. |

## Chinese Naming Review

The Chinese layer names must be reviewed separately from the architecture. A term can be architecturally accurate and still be bad public language if a first-time reader cannot understand it.

| Candidate term | Possible issue | Plain-language test | Status |
| --- | --- | --- | --- |
| 原书证据层 | `证据层` may sound legalistic or abstract. | Can a reader infer that this means preserving what the book says and where it came from? | Needs review before freeze. |
| 判断提炼层 | `提炼` may sound like summarization, while this layer means extracting usable judgment. | Can a reader infer that this means turning source material into transferable judgment rather than chapter notes? | Needs review before freeze. |
| 行动技能层 | Mostly clear, but must not imply every skill directly executes an action. | Can a reader understand that action includes judging, refusing, deciding, and recalibrating? | Likely acceptable. |
| 流程路由层 | Clear if paired with examples; abstract without them. | Can a reader understand that deterministic workflows stay separate from judgment skills? | Likely acceptable with examples. |
| 应用校准层 | Better than `世界对齐层`, but `校准` may still be abstract. | Can a reader infer that this means applying current context without rewriting the source skill? | Needs review before freeze. |
| 行动价值验收 | Accurate for engineering; may sound too internal for README. | Should public docs use `行动价值评估` while release gates use `行动价值验收`? | Needs review before freeze. |

Chinese naming acceptance criteria:

- A term should explain itself with at most one short sentence.
- A term should not require knowing historical KiU releases or external reference projects.
- A term should be understandable in README headings, not only in engineering reports.
- If a term is accurate but stiff, the design doc must record a plainer public alias.
- Do not freeze the glossary until this review chooses final Chinese names and aliases.

## Language Rules From v0.8 Onward

- New mainline docs must use the five-layer architecture names above.
- Historical terms may appear in historical evidence, attribution, or the glossary only.
- When a historical term must appear in a new explanatory doc, introduce it once as history, then continue with the v0.8 term.
- Do not mass-edit old reports or plans just to rename terms.
- Do not use language simplification to weaken evidence honesty or workflow-vs-agentic boundaries.
```

Add this document:

```markdown
# Project Architecture Narrative

KiU (`学以致用`) turns source knowledge into usable judgment.

It is not a summary engine, a quote database, a translator, or a generic RAG notebook. Those systems help users retrieve or restate knowledge. KiU tries to answer a harder question: when should knowledge change how a user judges, chooses, refuses, acts, or recalibrates?

## The Short Version

```text
Book -> Evidence -> Judgment -> Skill or Workflow -> Application Calibration -> Action Value
```

The project keeps these steps separate because each step has a different failure mode.

## 1. 原书证据层

The source evidence layer preserves what the book says: passages, claims, structure, anchors, and provenance. This layer can support summaries and lookup, but it is not the final product.

## 2. 判断提炼层

The judgment distillation layer asks what transferable judgment the source supports. This is where KiU separates usable thinking from mere notes.

The key rule is: `技能不是摘要`.

## 3. 行动技能层

The action skill layer publishes only artifacts that can help a user judge, choose, refuse, act, or recalibrate. A published skill must know when it should fire and when it should not.

## 4. 流程路由层

Some useful source material is deterministic procedure. KiU keeps that material as workflow output instead of pretending it is a judgment skill.

## 5. 应用校准层

Application context can matter. Current facts, market conditions, user constraints, or risk level may change whether a skill should be applied.

The key rule is: `用而不染`. Application context may gate or caveat use, but it must not rewrite the source-derived skill.

## 6. 行动价值验收

KiU evaluates whether generated outputs help users in four action layers:

- Discover signals.
- Define problems.
- Resolve actions.
- Calibrate feedback.

Evidence levels remain explicit. Internal scenario checks are not real-user validation, and same-book reference comparisons are not external preference closure.

## Why This Architecture Exists

Without separation, a system tends to confuse five things: what the book said, what the book implies, what action a user should take, which deterministic workflow applies, and whether current-world facts make the action safe. KiU keeps those apart so that knowledge can be used without losing its source boundary.
```

# User-Facing Evaluation Guide

KiU has many internal checks, but users usually care about six questions.

| User-facing dimension | User question | Related internal evidence |
| --- | --- | --- |
| Source trust | Did the skill stay faithful to the source? | source fidelity, anchors, source pollution checks, extraction kind. |
| Action helpfulness | Does the skill help me judge, choose, refuse, act, or recalibrate? | action-value evaluation, usage review, action-skill identity. |
| Boundary clarity | Does the skill know when not to fire? | anti-conditions, misuse checks, should-not-trigger scenarios. |
| Coverage fit | Does the visible output cover the book's useful value? | coverage readiness, skill/workflow routing, source-value routes. |
| Context application safety | Does application context help without rewriting the source? | application calibration, current-fact checks, source pollution checks. |
| Evidence confidence | What kind of proof supports the claim? | internal review, proxy checks, same-book comparison, blind review, real-user evidence. |

## Evidence Confidence Levels

| Level | Meaning | Claim allowed |
| --- | --- | --- |
| Internal artifact review | Project team reviewed generated artifacts. | Internal quality signal. |
| Internal scenario check | Synthetic or scripted scenarios passed. | Regression and pressure evidence. |
| Same-book reference comparison | KiU compared against a separate reference on the same book where possible. | Comparative signal, not final preference proof. |
| External blind review | Outside reviewer chooses without attribution leakage. | External preference evidence. |
| Real-user validation | Real users apply skills in real decisions. | Usage-value evidence. |
| Domain-expert review | Qualified expert reviews source fidelity and action safety. | Expert confidence evidence. |

## Reading A Scorecard

A high score means the current evidence supports the claim at the stated evidence level. It does not automatically upgrade internal evidence into external validation.

For example, a score of `94` under internal artifact review means the generated artifact reads strongly to the project team. It does not mean real users prefer it until real-user or external blind evidence exists.
```
