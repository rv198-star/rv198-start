# v0.8.1 Repository Structure Cleanup Design

## Goal

v0.8.1 should make the repository structure easy to understand for new reviewers and future development work by separating current project entry points, current reviewer artifacts, engineering assets, historical evidence, examples, and archived process material.

The goal is structural clarity, not new product capability.

## Current Baseline

After v0.8.0, the repository successfully exposes the project narrative, methodology toolbox, and `review-pack/current/`. The remaining problem is that top-level and documentation directories still mix several roles:

- `docs/` contains current public docs, method docs, implementation plans, historical planning notes, audit files, and old design records.
- `reports/` contains release evidence, historical audit reports, JSON gate evidence, blind-review packs, and old process outputs in one flat surface.
- `examples/` contains source books, fixtures, legacy source bundles, and public example materials.
- `notion-export/` is early migration material and should not remain a first-class project directory.
- `bundles/` contains legacy sample bundles that are still useful for tests and compatibility, but no longer represent the current reviewer-facing output path.
- `review-pack/current/` is now the canonical current artifact review entry and should remain visible.

The current structure makes the project look more complicated than it is because current and historical materials have the same visual weight.

## Target Outcome

Root-level structure should communicate five things quickly:

1. How to understand the project.
2. How to run or develop the code.
3. How to inspect current generated artifacts.
4. Where current and historical evidence live.
5. Where legacy or process material has been archived.

## Directory Responsibilities

| Path | Responsibility |
| --- | --- |
| `src/` | Python package source code. |
| `scripts/` | CLI and developer scripts. |
| `schemas/` | Machine-readable schemas. |
| `tests/` | Automated tests. |
| `shared_profiles/` | Current profile configuration. |
| `backlog/` | Current backlog board. |
| `docs/public/` | First-time-reader and public project explanation. |
| `docs/methodologies/` | Methodology toolbox; keep as current. |
| `docs/engineering/` | Specs, usage guides, evaluation templates, and engineering-facing documentation. |
| `docs/release-history/` | Version plans, review responses, and roadmap records that are still useful for release traceability. |
| `docs/archive/` | Historical context that should remain traceable but is not a current entry point. |
| `evidence/current/` | Current release evidence and closeout evidence. |
| `evidence/archive/` | Historical reports and old blind-review packs. |
| `review-pack/current/` | Current reviewer-facing source and generated artifact pack. Keep visible and current-only. |
| `examples/fixtures/` | Test fixtures and tiny examples used by tests. |
| `examples/source-materials/` | Source material examples used to demonstrate or run the pipeline, not the canonical current review source. |
| `examples/legacy/` | Old examples retained for traceability. |
| `bundles/` | Legacy example bundles used by tests and compatibility examples; keep root path stable and label it clearly. |
| `_archive/notion-export/` | Raw Notion export material, downgraded from first-class project structure. |

## Non-Goals

- Do not change generation behavior.
- Do not regenerate artifacts.
- Do not modify historical report content just to update old links.
- Do not remove `review-pack/current/`.
- Do not claim external validation from internal evidence.
- Do not delete historical evidence unless it is clearly local-only cache material or duplicated by current review pack and not referenced.

## Link Policy

Update only active entry points and tests:

- `README.md`
- `AGENTS.md`
- `CHANGELOG.md`
- `docs/README.md`
- `evidence/README.md`
- `examples/README.md`
- `bundles/README.md`
- automated tests that check paths

Historical reports may keep historical paths. This preserves provenance and avoids creating false history.

## Verification Plan

Run staged verification after migration:

1. Structure checks:

```bash
test -f docs/README.md
test -f evidence/README.md
test -f review-pack/current/README.md
test -d docs/public
test -d docs/engineering
test -d docs/release-history
test -d evidence/archive
```

2. Link and terminology checks:

```bash
rg -n "docs/project-architecture-narrative.md|docs/concept-language-glossary.md|docs/user-facing-evaluation-guide.md" README.md AGENTS.md docs/README.md docs/public docs/engineering tests || true
rg -n "reports/|notion-export/" README.md AGENTS.md docs/README.md docs/public docs/engineering tests || true
```

Historical docs under `docs/release-history/`, `docs/archive/`, and `evidence/archive/` are allowed to contain old paths.

3. Test subsets:

```bash
PYTHONPATH=src .venv/bin/python -m unittest tests.test_packaging tests.test_backlog -v
PYTHONPATH=src .venv/bin/python -m unittest discover tests -v
```

If full discovery is too slow or noisy, run affected path-sensitive tests first and report any skipped full-suite scope honestly.

## Risks

| Risk | Mitigation |
| --- | --- |
| Broken README or AGENTS links | Update active links and run `rg` checks. |
| Tests assume old `examples/` or `bundles/` paths | Update tests or preserve compatibility README/path where cheaper. |
| Large move creates noisy diff | Use `git mv`; split implementation into batches. |
| Historical reports become misleading if edited | Do not rewrite old reports except index/navigation files. |
| Review pack gets hidden | Keep `review-pack/current/` at root and link from README. |

## Proposed Implementation Batches

Batch 1: Add structure indexes and non-disruptive directories.

- Add `docs/README.md`, `evidence/README.md`, `review-pack/README.md`, `bundles/README.md`.
- Add `docs/public/`, `docs/engineering/`, `docs/release-history/`, `docs/archive/`, `evidence/current/`, `evidence/archive/`.
- Update README links to planned current locations only after files move.

Batch 2: Move current/public and engineering docs.

- Move current public docs into `docs/public/`.
- Move engineering docs into `docs/engineering/`.
- Update README, AGENTS, and path-sensitive tests.

Batch 3: Move historical evidence and process material.

- Move `reports/` content into `evidence/current/` and `evidence/archive/`.
- Move `notion-export/` into `_archive/notion-export/`; do not delete it in v0.8.1.
- Move `examples/` source material into `examples/source-materials/`, move `workflow_candidates/examples/` into `examples/legacy/workflow-candidates/`, and add `bundles/README.md` while keeping bundle paths stable.

Batch 4: Final audit.

- Run link checks.
- Run path-sensitive tests.
- Run broader test discovery if feasible.
- Add a short v0.8.1 repository-structure evidence note if needed.

## Acceptance Criteria

- Root directory no longer presents `reports/` and `notion-export/` as first-class current project surfaces.
- `README.md` points to current project explanation, current review pack, methodology toolbox, and evidence index.
- `docs/` separates public/current docs from historical plans and engineering docs.
- `evidence/` becomes the top-level home for current and archived evidence.
- `review-pack/current/` remains the canonical reviewer artifact entry.
- `examples/` no longer mixes current review sources, fixtures, and legacy examples without labels.
- Tests pass for affected packaging/path checks.
- Any skipped or deferred cleanup is recorded explicitly rather than hidden.
