# v0.7.2 Live Fact Verification Design

## Version Goal

v0.7.2 introduces true live web retrieval for bounded current-world fact verification.

The version goal is not broad Web-RAG, not continuous real-time enrichment, and not rewriting source-derived skills with external information. It adds an isolated live-fact layer that can verify or reject specific current-world claims when a user request depends on facts that the offline v0.7.0/v0.7.1 world-alignment model cannot safely assert.

Core claim:

```text
v0.7.2 verifies bounded current-world claims with cited live sources when live check is triggered. Verified facts may gate application, but may not rewrite source skills or become source evidence.
```

## Claim Boundary

v0.7.2 may claim:

- Bounded current-world claim detection from existing no-web hallucination and temporal-sensitivity signals.
- Live retrieval for specific claims that require current facts.
- Citation-backed fact packs with URL, title, retrieval time, source date when available, and relation-to-claim labels.
- Application-gate behavior that refuses, asks for context, or adds caveats when facts are unsupported, conflicting, stale, or insufficient.
- Source pollution prevention between original-book evidence and external world evidence.

v0.7.2 may not claim:

- Exhaustive factual correctness.
- Universal currentness.
- Domain-expert validation.
- Real-user validation.
- Replacement of human judgment in high-stakes legal, medical, financial, or safety domains.
- That live facts are source evidence for the original book.
- That `SKILL.md` is rewritten or improved by external facts.

## AGENTS Alignment

### External Reference Boundary

Live web sources are external context, not hidden upstream generators and not source evidence. They must be stored under isolated world-alignment artifacts and must not be silently merged into the default production skill-generation path.

### Workflow-vs-Agentic Boundary

Retrieval itself is a workflow operation: query, fetch, capture metadata, and write evidence. Claim support judgment is a bounded agentic verification step. Application decisions remain judgment-rich gate behavior. v0.7.2 must not create an unbounded web-browsing agent.

### System Efficiency Over Local Advantage

The efficient path is claim-level verification, not broad web-context generation. Low-temporal mental models should not trigger web checks by default. High-temporal current-fact claims should trigger live verification or degrade to ask/refuse/caveat.

## Operating Modes

### `source_only`

Only source-faithful skills and original book evidence are used. No world-alignment enhancement and no live fact verification are applied.

### `no_web`

Offline world alignment may run, but current-world factual claims are blocked when they require live verification. Source-faithful skills remain usable. High-temporal direct application must not fabricate current facts.

Correct behavior:

```text
No live web => no verified current-world application.
```

### `live_web_allowed`

Live verification runs only when a claim ledger item requires it. Low-temporal skills may still bypass live retrieval. Verified facts can affect application-gate verdicts, caveats, and clarification requests.

### `live_web_required`

The request depends on current-world facts. If retrieval fails, the application gate must refuse direct application or ask for current inputs. It must not fall back to unsupported model knowledge.

### `retrieval_failed`

Network or retrieval failure is an explicit state. The system may still answer source-faithful or principle-level parts, but must mark current-world application as unverified and avoid pretending that the claim was checked.

## Architecture

### Claim Ledger

`claim_ledger.py` bridges the v0.7.0/v0.7.1 no-web hallucination and temporal-sensitivity layer into structured verification tasks.

Output:

```yaml
schema_version: kiu.claim-ledger/v0.1
claims:
  - claim_id: claim-001
    text: "当前市场数据证明这家公司被低估"
    claim_type: current_market_fact
    origin: user_prompt
    temporal_sensitivity: high
    verification_required: true
    no_web_status: blocked_current_fact_claim
    allowed_without_web: false
    verification_status: pending
```

### External Fact Pack

`live_facts.py` performs minimal retrieval and writes evidence. It does not produce advice and does not alter source artifacts.

Output:

```yaml
schema_version: kiu.external-fact-pack/v0.1
retrieved_at: "2026-04-25T00:00:00Z"
retrieval_mode: live_web
facts:
  - claim_id: claim-001
    claim: "当前市场数据证明这家公司被低估"
    verification_status: unsupported
    evidence:
      - source_url: "https://example.com/source"
        source_title: "Source title"
        published_at: "2026-04-20"
        retrieved_at: "2026-04-25T00:00:00Z"
        relation_to_claim: related_but_not_supporting
    freshness_status: current
    confidence: low
    decision_effect:
      - refuse_direct_application
      - ask_for_current_valuation_inputs
```

### Claim Verification

`fact_verification.py` compares retrieved evidence against each claim and assigns one of the following statuses:

- `supported`
- `partially_supported`
- `unsupported`
- `conflicting`
- `stale`
- `undated`
- `insufficient_evidence`
- `retrieval_failed`

Only `supported` and narrowly scoped `partially_supported` claims may enable direct application, and even then with visible caveats when the domain is high-stakes.

### Freshness Gate

`freshness_gate.py` maps verification status to application-gate behavior.

Required mappings:

- `supported` => allow application with citation caveat.
- `partially_supported` => partial apply or ask for missing context.
- `unsupported` => refuse direct current-world application.
- `conflicting` => refuse direct application and expose conflict.
- `stale` => ask for fresher source or user-provided current data.
- `undated` => caveat or ask unless the claim is low temporal sensitivity.
- `insufficient_evidence` => ask/refuse, no fabrication.
- `retrieval_failed` => no verified current-world application.

### World Alignment Consumer

Existing world-alignment code consumes claim-ledger and fact-pack outputs. It may change application verdicts and caveats, but it must not mutate `SKILL.md` or source-derived evidence.

## Release Gates

v0.7.2 has two mandatory gates.

### Gate 1: Deterministic Fixture Gate

All fixture checks must pass:

- `deterministic_fixture_matrix_passed = true`
- `no_web_hallucination_regression = pass`
- `citation_completeness >= 95%`
- `relation_to_claim_completeness >= 95%`
- `source_pollution_errors = 0`
- `fact_pollution_errors = 0`
- `network_failure_fabrication_count = 0`
- `high_temporal_live_check_trigger_rate >= 90%`
- `low_temporal_live_check_suppression_rate >= 90%`
- `unsupported_claim_direct_apply_count = 0`
- `conflicting_fact_direct_apply_count = 0`
- `stale_current_claim_direct_apply_count = 0`

### Gate 2: Real Live Retrieval Gate

Live retrieval is a hard release gate, not an optional smoke test:

- `real_live_retrieval_success_count >= 3`
- `real_live_claim_verification_success_count >= 2`
- `real_live_citation_completeness = 100%`
- `real_live_network_failure_fabrication_count = 0`
- `real_live_external_fact_pack_written = true`

The live gate should initially prefer official, public, stable, date-bearing sources. Fragile real-time prices may be used only as refusal/ask cases unless the source and retrieval path are reliable enough for repeatable evidence.

### Gate 3: Value Delta And Regression Attribution

The live-fact layer must not pass release only by proving that retrieval works. It must also report whether live verification improved or degraded practical application behavior.

Required evidence:

- `value_delta_and_regression_attribution_written = true`
- Same-run live-on/live-off or supported/unsupported comparison exists for at least the deterministic fixture set.
- Same-book reference comparisons are labeled as same-book unless strict source-data parity is proven.
- Any benchmark delta narrowing is attributed to scenario-set changes, stricter gate behavior, proxy scoring, or unresolved unknowns rather than silently treated as success.
- Real samples and fixtures are counted separately.
- Mechanism counters state their grain and include residual buckets.

This gate inherits the v0.7.1 audit addendum rule: release-facing evidence must show actuals, thresholds, statuses, and evidence paths in one place.

## Ticket Plan

### KIU-730: Claim Ledger And No-Web Bridge

Create a claim ledger that converts no-web hallucination detections and high-temporal application requests into structured verification tasks.

Acceptance:

- Claim ledger schema exists and is documented.
- High-temporal current-world claims become `verification_required=true`.
- Low-temporal source-faithful applications are not forced into live retrieval.
- No-web behavior blocks direct current-fact application without disabling source-faithful skills.

### KIU-731: External Fact Pack Schema

Define the isolated fact-pack artifact for live retrieval outputs.

Acceptance:

- Fact pack records source URL, title, retrieved time, published date when available, relation to claim, freshness status, confidence, and decision effect.
- Fact pack is stored under `world_alignment/` or reports, not under source evidence.
- Missing citation metadata fails validation unless explicitly marked unavailable.

### KIU-732: Live Retrieval Adapter

Implement minimal live retrieval for bounded claim checks.

Acceptance:

- Adapter fetches live evidence for specific claims.
- Retrieval records query, source URL, title, retrieved time, and raw support snippet or summary.
- Retrieval failure is represented explicitly and never converted into model-invented evidence.

### KIU-733: Claim-Level Fact Verification

Classify the relationship between evidence and claim.

Acceptance:

- Verification statuses cover supported, partial, unsupported, conflicting, stale, undated, insufficient evidence, and retrieval failure.
- Unsupported, conflicting, stale, insufficient, and retrieval-failed claims cannot directly apply.
- Tests include positive and negative examples for each high-risk status.

### KIU-734: Application Gate Fact Consumption

Make application gates consume verification results.

Acceptance:

- Supported facts can enable cited application.
- Partial support produces partial apply or ask-more-context.
- Unsupported/conflicting/stale/retrieval-failed claims produce refuse/ask/caveat behavior.
- `SKILL.md` remains unchanged.

### KIU-735: Hallucination Regression Matrix

Extend existing no-web hallucination fixtures into live/no-live regression tests.

Acceptance:

- Positive live-supported cases pass.
- Negative unsupported/conflicting/stale cases are blocked.
- No-network cases do not fabricate current facts.
- Both source-only and no-web modes remain usable for source-faithful skills.

### KIU-736: Source Pollution And Citation Preflight

Add preflight checks that separate source evidence from live fact evidence.

Acceptance:

- Live URLs do not appear in source-derived `SKILL.md` evidence anchors.
- Original book source anchors do not get replaced by external citations.
- Fact packs fail validation when citation or relation-to-claim metadata is incomplete.

### KIU-737: v0.7.2 Live Freshness Evidence Pack

Produce release evidence for deterministic fixtures and real live retrieval.

Acceptance:

- Evidence pack reports all release gates with actuals, thresholds, pass/fail, and artifact paths.
- Live run evidence includes retrieved time and source list.
- Network failures, if any, are reported without being hidden.
- Evidence pack includes `value_delta_and_regression_attribution`, explaining live-on/live-off value deltas, same-book benchmark movement if present, real-vs-fixture sample counts, and any unresolved attribution limits.

### KIU-738: Live Check Claim Boundary And Release Notes

Document the release claim boundary.

Acceptance:

- Release notes state what v0.7.2 can and cannot claim.
- Release notes distinguish fixture validation from real live retrieval evidence.
- High-stakes domains retain human-judgment caveats.

### KIU-739: Real Live Retrieval Release Gate

Make live retrieval a hard release gate for v0.7.2.

Acceptance:

- At least three live retrievals succeed.
- At least two claim verifications succeed from live evidence.
- Citation completeness is 100% for live evidence.
- Network failure fabrication count is zero.
- The external fact pack is written and archived.

## Non-Goals

- No broad web crawling.
- No multi-world or multi-stance modeling.
- No always-on live enrichment for all skills.
- No real-time investment advice engine.
- No external blind review gate.
- No mutation of source-faithful `SKILL.md` artifacts.
- No hidden ingestion of external final skill packs.

## Open Risks

- Network access may be unavailable in some execution environments. This must produce explicit retrieval-failed behavior, not fabricated facts.
- Live sources can change. Evidence packs must include retrieval time and should prefer stable official sources for release gates.
- Verification quality depends on bounded claim formulation. Over-broad claims must be split or rejected.
- Real live validation increases reproducibility risk. Deterministic fixtures remain necessary as the stable regression base.
