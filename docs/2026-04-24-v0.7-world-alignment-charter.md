# KiU v0.7 World Alignment Charter

Date: `2026-04-24`

## Version Name

`KiU v0.7 · Temporal-Aware World Alignment`

中文定位：`时效感知的真实世界对齐层`。

## Core Goal

`v0.7` adds an isolated world-alignment layer over source-faithful KiU skills.

The goal is to make generated skills more useful under realistic use pressure while preserving:

- source fidelity
- evidence honesty
- workflow-vs-agentic boundary discipline
- auditable decision paths
- scalable system-level production quality

The version does not make external world information a hidden upstream generator. It introduces a reviewer / gate / explainer layer that can pressure-test and contextualize source-derived skills without silently rewriting them.

## Non-Goal

`v0.7` is not a general runtime agent, web-search product, or external fact-ingestion pipeline.

It should not attempt to solve every live-world fact problem in the first release. Runtime use-time context remains a compatible future input surface, but the first release should prioritize auditable bundle artifacts and review gates.

## Alignment With `AGENTS.md`

### External Reference Boundary

World context is not source evidence.

External reality, current facts, user-provided situation data, benchmark references, or simulated pressure cases may be used as world-alignment context, but they must not become hidden source anchors for the default skill-generation path.

Allowed effects:

- add usage caveats
- add realistic pressure tests
- add temporal context requirements
- gate application when current context is missing
- explain decision paths

Forbidden effects:

- rewrite source-derived `SKILL.md` rationale
- replace book/source anchors
- invent authorial claims
- silently widen a skill beyond the source-supported boundary
- ingest external final skill packs as default production inputs

### Workflow-vs-Agentic Boundary

World alignment must not improve apparent usefulness by collapsing deterministic workflow candidates into thick skills.

If a realistic use case only needs a stable procedure and explicit context, it remains a workflow candidate. If it requires judgment-rich interpretation, context assembly, or boundary arbitration, it may be evaluated by an agentic skill or by the world-alignment gate.

Release review must keep scoring workflow-vs-agentic boundary quality explicitly.

### System Efficiency Over Local Advantage

`v0.7` should favor repeatable, auditable, average-case improvement over handcrafted showcase examples.

World alignment should become a system layer with schemas, gates, and review scores, not a manual rewrite pass that produces a few locally excellent but non-scalable artifacts.

## Architectural Principle

World alignment is one temporal-aware context layer, not two separate release-time and runtime systems.

All external reality enters the same model as `world_context` items. A context item may be release-baseline, user-supplied, external-check-required, simulated, or runtime-observed. The handling is determined by metadata, not by separate pipelines.

The core objects are:

1. `world_context`
2. `temporal_policy`
3. `application_gate`
4. `pressure_tests`
5. `decision_path`
6. `scorecard`

## Proposed Bundle Surface

```text
bundle/
  skills/
    <skill_id>/
      SKILL.md
      anchors.yaml
      eval/
      iterations/
      usage/

  world_alignment/
    world_context.yaml
    temporal_profile.yaml
    application_gate_schema.yaml

    <skill_id>/
      temporal_policy.yaml
      pressure_tests.yaml
      usage_caveats.yaml
      decision_path.md
      scorecard.yaml

  traces/
    world_context/
      <case_id>.yaml
```

`skills/` remains source-faithful. `world_alignment/` is the only place where external reality, temporal caveats, pressure tests, and application-gate decisions may be stored.

## World Context Model

A world-context item has time and freshness attributes, but release-time and use-time items share the same schema.

Example:

```yaml
id: ai-prototyping-changed-requirements-work
context_type: release_baseline
observed_at: 2026-04-24
valid_until: null
freshness_policy: on_minor_release
temporal_sensitivity: medium
confidence: medium
source_summary: AI prototyping, low-code delivery, and compliance expectations can change how requirements-analysis methods are applied in modern organizations.
applies_to:
  - solution-to-problem-reframing
  - workflow-gateway
allowed_effects:
  - add_usage_caveat
  - add_pressure_test
  - require_current_context
forbidden_effects:
  - rewrite_source_rationale
  - replace_source_evidence
```

Use-time or runtime-compatible example:

```yaml
id: named-company-current-regulatory-status
context_type: external_check_required
observed_at: runtime_required
freshness_policy: before_use
temporal_sensitivity: high
required_when:
  - named_current_company
  - regulatory_or_compliance_claim
allowed_effects:
  - gate_application
  - ask_more_context
forbidden_effects:
  - rewrite_source_rationale
  - replace_source_evidence
```

## Temporal Sensitivity Strategy

Temporal alignment strength should be proportional to temporal sensitivity and application risk.

### Low Sensitivity

Long-lived principles and mental models.

Default handling:

- release-baseline world context optional
- pressure tests focus on misuse and boundary drift
- use-case context required only when the user names current entities or irreversible real-world decisions

### Medium Sensitivity

Stable principles applied in changing environments.

Default handling:

- release-baseline world context required
- use-case context required for concrete current organizations, assets, systems, regulations, technologies, or market conditions
- missing context usually yields `ask_more_context` or `apply_with_caveats`

### High Sensitivity

Domains where concrete application depends on current facts.

Default handling:

- release-baseline context required
- before-use context mandatory for application
- missing context yields `ask_more_context` or `refuse`
- source-derived skill may remain intact, but application is gated

## Application Gate

The application gate is the final world-alignment decision surface. It does not rewrite the skill; it decides whether the skill can be applied under the available context.

Allowed verdicts:

```yaml
verdict: apply | apply_with_caveats | partial_apply | ask_more_context | refuse
source_skill_unchanged: true
reason: string
required_context: list[string]
decision_path: list[string]
```

The gate must preserve a visible path:

1. source skill and source anchors
2. relevant world context items
3. temporal-policy check
4. application verdict
5. caveats or missing-context requirements

## Required Scores

`v0.7` introduces world-alignment scoring without replacing existing source/generated/usage review.

Recommended dimensions:

- `source_fidelity_score`
- `world_context_quality_score`
- `temporal_policy_fit_score`
- `application_gate_quality_score`
- `decision_path_clarity_score`
- `workflow_boundary_resilience_score`

Initial release gate:

```yaml
v0.7_release_gate:
  source_fidelity_preserved: true
  world_context_isolated: true
  workflow_boundary_preserved: true
  samples_passed_min: 3
  world_alignment_score_min: 85
  source_pollution_errors: 0
  application_gate_cases_min: 9
```

## Sample Scope

`v0.7` should be verified on at least three independent samples:

- Poor Charlie: low / medium temporal sensitivity, principle-heavy skills
- Effective Requirements: medium temporal sensitivity, method and organization-context skills
- Financial Statement: medium / high temporal sensitivity, concrete application and current-context gates

Each sample should prove:

- original `skills/` are not polluted by world context
- `world_alignment/` artifacts exist independently
- at least two skills have temporal policy
- at least three pressure tests cover realistic misuse or changing constraints
- application gate can produce at least three verdict types across the sample set
- review separates source fidelity from world usefulness

## Release Claim

A successful `v0.7` release may claim:

- KiU can keep source-derived skills stable while adding a separate world-alignment layer.
- KiU can explain when a skill is applicable, stale, risky, incomplete, or blocked by missing current context.
- KiU can improve practical usefulness without using external world context as hidden source evidence.

It may not claim:

- fully automated live-world factual correctness
- a complete runtime world-checking product
- replacement of human or domain-expert judgment in high-stakes domains
- that source-derived skills are rewritten to be universally current
