# v0.8 Architecture Language Design

Date: 2026-04-26

v0.8 rebuilds KiU's public language around `学以致用`: turning source knowledge into executable action capacity while preserving source fidelity, boundaries, and user choice.

This is a language and architecture clarity release. It does not rename historical evidence and does not change generation behavior by itself.

## Core Architecture

The public architecture should read as a sequence of actions, not as a stack of internal nouns:

```text
读准原书 -> 提炼判断 -> 生成技能 / 分流流程 -> 校准应用 -> 验证行动价值
```

| Step | Public wording | Precise architecture meaning | What it does | What it must not do |
| --- | --- | --- | --- | --- |
| 1 | 读准原书 | Source fidelity and evidence preservation | Preserve what the source says, where it says it, and how evidence is anchored. | Do not treat source material as action guidance before it has action-ready judgment. |
| 2 | 提炼判断 | Judgment distillation | Extract transferable judgment from source evidence. | Do not reduce the book to summaries, quotes, or encyclopedia answers. |
| 3A | 生成技能 | Action skill publication | Publish skills that help users judge, choose, refuse, act, or recalibrate. | Do not publish source containers, chapter headings, exercises, or deterministic workflows as thick skills. |
| 3B | 分流流程 | Workflow routing | Keep deterministic procedures auditable as workflows and route users to them when appropriate. | Do not collapse judgment-heavy skills into checklists or inflate checklists into skills. |
| 4 | 校准应用 | Application calibration | Add isolated current context, fact checks, caveats, and safety gates when needed. | Do not rewrite source-faithful skills into blended application artifacts. |
| 5 | 验证行动价值 | Action-value evaluation | Check whether outputs help users discover signals, define problems, resolve actions, and calibrate feedback. | Do not claim real-user or external validation from internal evidence. |

## Core Public Phrases

`技能不是摘要` means KiU does not exist to summarize books. It turns source knowledge into bounded, triggerable action capacity.

`学到最后要能用` means learning is not complete until the source can improve concrete judgment, choice, action, refusal, or feedback.

These are the two preferred public phrases for v0.8. `用而不染` remains the architecture principle for 校准应用: application context may help users apply a skill, but it must not pollute or rewrite the source-derived skill.

## Coverage Review

| Current capability | v0.8 architecture home | Coverage judgment |
| --- | --- | --- |
| Source bundles, evidence anchors, provenance, source snapshots | 读准原书 | Covered. |
| Candidate extraction, distillation, seed verification, refinement loops | 提炼判断 | Covered. |
| Published `bundle/skills/*/SKILL.md` | 生成技能 | Covered. |
| Workflow candidates and gateway/router artifacts | 分流流程 | Covered. |
| Current context, fact checks, hallucination gates, application caveats | 校准应用 | Covered. |
| Method checks, source/fact checks, action-value scores, user-facing scorecards | 验证行动价值 | Covered. |
| Reference comparisons and blind-review packs | Evidence confidence inside 验证行动价值 | Covered as evidence tier, not core architecture. |

Coverage conclusion: the action-sequence architecture covers current KiU behavior without requiring old release vocabulary or external-reference vocabulary in the main explanation.

## Clarity Review

| Check | Result |
| --- | --- |
| Can a new reader explain the project in 90 seconds? | Yes: KiU turns books into executable action capacity, not summaries. |
| Does the architecture avoid excessive new jargon? | Yes: five action phrases plus two memorable rules. |
| Does it depend on knowing prior releases or reference projects? | No: historical vocabulary moves to attribution, archived evidence, and the glossary. |
| Does it preserve engineering boundaries? | Yes: source fidelity, workflow routing, and application isolation remain explicit. |
| Can it drive README structure? | Yes: the README can present problem, architecture, evidence boundary, then usage. |

Clarity conclusion: public docs should use the action phrases first. More precise internal names can appear after them when necessary.

## Chinese Naming Review

The Chinese names must be reviewed separately from the architecture. A term can be architecturally accurate and still be bad public language if a first-time reader cannot understand it.

| Earlier working term | Issue | v0.8 public wording | Decision |
| --- | --- | --- | --- |
| 原书证据层 | `证据层` is accurate but legalistic and abstract. | 读准原书 | Use the public wording in README and narrative docs; reserve evidence language for technical sections. |
| 判断提炼层 | Accurate but may sound like summarization. | 提炼判断 | Use the verb-object form to emphasize producing judgment, not summaries. |
| 行动技能层 | Clear enough, but may imply only direct execution. | 生成技能 | Use `生成技能` in architecture flow; explain that skill value includes judging, refusing, and recalibrating. |
| 流程路由层 | Reasonable but still internal. | 分流流程 | Use the simpler public wording; technical docs may still say workflow routing. |
| 应用校准层 | Better than broader alignment language, but still abstract. | 校准应用 | Use the verb-object form and treat `用而不染` as this step's boundary principle. |
| 行动价值验收 | Accurate for gates, stiff for public docs. | 验证行动价值 | Use `验证行动价值` publicly; use `行动价值验收` only for release gates and scorecards. |

Chinese naming acceptance criteria:

- A term should explain itself with at most one short sentence.
- A term should not require knowing historical KiU releases or external reference projects.
- A term should be understandable in README headings, not only in engineering reports.
- If a term is accurate but stiff, public docs should use the plainer action phrase and technical docs may record the precise alias.

Naming conclusion: v0.8 public language should prefer action phrases over noun-heavy layer names.

## Language Rules From v0.8 Onward

- New mainline docs should lead with: 读准原书, 提炼判断, 生成技能, 分流流程, 校准应用, 验证行动价值.
- Historical terms may appear in historical evidence, attribution, or the glossary only.
- When a historical term must appear in a new explanatory doc, introduce it once as history, then continue with the v0.8 term.
- Do not mass-edit old reports or plans just to rename terms.
- Do not use language simplification to weaken evidence honesty or workflow-vs-agentic boundaries.
- README should use the public action phrases first and only introduce precise engineering aliases after the reader understands the flow.
