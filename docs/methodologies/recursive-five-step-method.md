# 三层递归五步法

三层递归五步法是 KiU 的项目开发方法论。它用于复杂规划、审计反馈、版本目标、模糊工程决策和行动价值评估。

中文口诀：`基 -> 标 -> 差 -> 策 -> 拆`。

英文对应：`Baseline -> Target -> Gap -> Strategy -> Breakdown`。

## 核心升级

这个方法不只用于解决已经明确的问题。KiU 把同一套五步循环应用在三层：发现问题、定义问题、解决问题。

```text
发现层：世界 -> 信号
定义层：信号 -> 问题
解决层：问题 -> 行动
执行反馈 -> 回到发现层 / 定义层 / 解决层
```

## L1 发现层：世界 -> 信号

目标是从噪声中发现可复述、可定位、可重现的信号。

这一层的拆解输出不是解决方案，而是观察动作。

## L2 定义层：信号 -> 问题

目标是把信号转成可证伪、可度量的问题陈述。

这一层的拆解输出不是行动计划，而是精准的问题定义。

## L3 解决层：问题 -> 行动

目标是把已定义的问题转成可执行行动。

这一层的拆解输出必须包含输入、输出、执行者或执行代理、验证方式和验收标准。

## 每一层都走五步

| 步骤 | 动作 | 核心问题 |
| --- | --- | --- |
| 基线 | 定义现状事实、证据和限制。 | 我们现在在哪？哪些已证明，哪些只是猜测？ |
| 目标 | 定义期望终态和验收标准。 | 到什么状态才算这一层成功？ |
| 差距 | 找出现状与目标之间的具体距离。 | 缺的是证据、能力、资源、路径还是认知？ |
| 策略 | 选择跨越差距的路线。 | 当前约束下哪条路径最划算？哪些路暂时不走？ |
| 拆解 | 输出当前层级需要的交付物。 | 这个叶子节点现在能观察、能判断或能执行吗？ |

## 递归规则

如果任何子问题还不能被观察、判断或执行，就不要把它伪装成行动项，而是对它再走一轮 `基 -> 标 -> 差 -> 策 -> 拆`。

递归终止条件：叶子节点已经变成观察动作、可证伪问题陈述或可执行行动，并且有足够的输入、输出、验证方式和验收标准。

## 决策检查

- 我们是在发现真实信号，还是在解决最显眼的症状？
- 我们是在定义可证伪问题，还是在复述偏好、抱怨或结论？
- 我们是在真实基线上解决问题，还是在假设状态上解决问题？
- 目标是否足够可衡量，能够拒绝虚假完成？
- 策略是明确取舍，还是一串愿望清单？
- 每个行动是否能直接启动并被判断？
- 不可执行的子问题是否已经递归，而不是被伪装成任务？

## 边界

- 不覆盖证据诚实。
- 不覆盖流程/判断边界。
- 不覆盖应用校准隔离。
- 不覆盖外部参考边界。
- 小任务可以压缩使用，但顺序不能乱：先基线，再目标，再差距，再策略，最后拆解。
