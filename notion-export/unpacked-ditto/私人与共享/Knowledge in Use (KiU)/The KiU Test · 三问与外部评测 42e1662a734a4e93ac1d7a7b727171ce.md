# The KiU Test · 三问与外部评测

<aside>
🚪

**KiU Test 是入库的唯一门槛**。不过关的候选 → `rejected/`,不讨价还价。

</aside>

## 三问 (机器可判定)

### Q1 · Trigger Test

> 这条 skill 能告诉我 "什么情境下我该调用它" 吗?
> 

**判定方式**:

- `use.trigger.patterns` 存在且 ≥1 条
- 每条 pattern 必须是机器可匹配的 pattern_id (snake_case, 不含特殊字符)
- 不允许 "读一读然后自己判断" 这样的开放表述

**不过示例**:

```yaml
trigger:
  patterns:
    - "when the user is thinking about investing"  # ❌ 散文, 不可机器匹配
```

### Q2 · Fire Test

> 给定一个符合 trigger 的新情境,这条 skill 能输出具体判断吗?
> 

**判定方式**:

- `use.judgment_schema.output` 定义明确
- schema 不能是纯文本或者指向原文
- `reasoning_chain_required: true` 时必须产出推理链

**不过示例**:

```yaml
judgment_schema:
  output:
    type: string
    schema: "参考原书第X章"  # ❌ 指向原文, 不是判断
```

### Q3 · Boundary Test

> 给定一个故意错配的情境,它会主动拒绝触发吗?
> 

**判定方式**:

- `use.boundary.fails_when` ≥1 条
- `use.boundary.do_not_fire_when` ≥1 条
- 每条必须是机器可匹配的模式,不能是"小心使用"这种空洞警告

**不过示例**:

```yaml
boundary:
  fails_when:
    - "如果情况特殊时请谨慎"  # ❌ 空洞, 没有情境挺定性
  do_not_fire_when: []         # ❌ 空列表
```

## External Evaluation Corpus · 验证铁底

**KiU Test 三问是结构校验。三问通过 ≠ skill 有用。**最后要在**语料外的真实决策**上走一遍:

```
evaluation_corpus/
├── real_decisions/           带已知结果的真实决策
│                            (年报、访谈、论坛、公开持仓变动历史)
├── synthetic_adversarial/    人工构造的陷阱情境
│                            (领域专家出题,测 boundary)
└── out_of_distribution/      故意的领域外情境
                             (测 skill 是否会主动拒绝触发)
```

## Evaluation Gate 通过标准

| 子集 | 期望行为 | 通过阈值 (v0.1) |
| --- | --- | --- |
| real_decisions | skill 输出与实际结果一致率 | ≥ 70% (或 ≥ 指定 baseline) |
| synthetic_adversarial | 对陷阱情境输出 NOT_APPLICABLE | ≥ 85% |
| out_of_distribution | skill 主动不触发 | ≥ 90% (这是 boundary 的实质测试) |

**任何子集不达标 → skill 不进 skill pack**。进入 `candidates/` 归档,等 L5 loop 打 patch。

## Gate 流程图

```
      候选 skill
         ↓
┌───────────────────┐
│ KiU Test 三问       │  ← 结构校验 (下秒级)
└────┬───────────────┘
     ↓ Pass
┌───────────────────┐
│ Internal Self-Test  │  ← 用 test-prompts.json 自测
└────┬───────────────┘
     ↓ Pass
┌───────────────────┐
│ External Eval Gate  │  ← 真实数据,分试集考核
└────┬───────────────┘
     ↓ Pass
  published skill
```

全部过 → 进 `skills/`。  

任一没过 → 进 `candidates/`,附失败原因,等 L5 执行 gap analysis。

## 为什么这三问就够

三问在结构上覆盖了 "可执行" 的全部必要条件:

| 三问 | 对应的可执行属性 |
| --- | --- |
| Trigger | 切入条件 (没有这个,skill 永远等待不到自己的场景) |
| Fire | 具体判断产出 (没有这个,使用者还得自己推理) |
| Boundary | 失效模式声明 (没有这个,skill 会在不该触发的时候起灵) |

三问又不容易被滥用 —— 每一个都需要写出具体的结构化 pattern,不能用泛泛的散文蒙混过关。
