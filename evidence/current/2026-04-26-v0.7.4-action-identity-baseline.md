Source runs: `/tmp/kiu-v072-readiness2-eval/*/generated/*/*/bundle`
Mode: diagnostic only; this baseline does not change publication behavior.

## Purpose

v0.7.3 introduced C-class action-value metrics and exposed a failure mode: a generated artifact can score well on internal structure while failing to become a triggerable action skill. v0.7.4 therefore starts with an explicit `action-skill identity` diagnostic before changing routing behavior.

Definition:

```text
action-skill identity = whether a candidate has transformed source value into a triggerable judgment/action capability installable as KiU SKILL.md
```

This diagnostic intentionally distinguishes:

```text
source_value != action_value
action_value != action_skill_identity
```

## Cross-Book False Positive Check

The first diagnostic draft overmatched incidental words such as `concept`, `summary`, and `exercise` inside otherwise valid SKILL.md bodies. The model was corrected to treat container identity as a label-level signal, not a body-level incidental word. After that correction:

- Poor Charlie has `false_container_positive_count = 0`.
- Shiji has `false_container_positive_count = 0`.
- Mao's `no-investigation-no-decision` and `principal-contradiction-focus` route to `publish_skill`.
- Effective Requirements keeps `solution-to-problem-reframing` and `stakeholder-resistance-tradeoff` as `publish_skill` and `workflow-gateway` as `publish_gateway`.

## Interpretation

The issue is pipeline-level, not a Financial Statement-specific naming bug. The generation path lacked an explicit identity gate between candidate mining and publication. When source materials contained structured containers such as exercises, concept sections, toolboxes, or case libraries, the existing structure-quality checks could accept them as published skills because they were evidence-rich and internally coherent.

Batch 2 should therefore add a semantic candidate-routing gate:

- `publish_skill` for thick judgment/action skills with `action_skill_identity_score >= 0.85`.
- `publish_gateway` for thin workflow routers with explicit gateway boundary evidence.
- `route_workflow_candidate`, `route_concept_note`, `route_case_library`, `route_evaluation_material`, or `route_source_context` for non-skill source value.

The fix must remain model/routing based and must not edit generated SKILL.md artifacts directly.
